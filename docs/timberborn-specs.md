# Timberborn ComponentSpec to runtime-component reference

## What this doc is

A lookup reference for what runtime `BaseComponent`s vanilla Timberborn 1.0
attaches to a prefab in response to which `ComponentSpec`s being present
in its blueprint JSON. Built by extracting every `AddDecorator<TSpec, TComponent>()`
call from the 194 vanilla configurators that participate in the
`TemplateModule` pipeline — 1033 decorator pairs in total — plus the
non-decorator prefab-converter pipeline that handles `CollidersSpec`,
`TransformSpec`, and `TimbermeshSpec`.

The motivating use case is Keystone-authored blueprint debugging: when a
custom entity composes a stack of vanilla specs and *behaves* wrong (not
selectable, mark-for-destruction tool ignores it, entity panel is blank,
no selection sound, beavers won't path to it…), the diagnosis nearly
always reduces to "you forgot to compose one of the specs that wires up
the missing seam." This doc lets you look that up without re-decompiling
DLLs every time.

Cross-references:
- `docs/timberborn-api-full.md` — autogenerated full type/signature dump
  for all 490 Timberborn DLLs. Use for "does service X expose method Y."
- `docs/timberborn-api.md` — hand-curated companion: two-axis API
  patterns, lifecycle ordering, blueprint-overlay system, sub-tile
  composition with `TransformSpec`. Use for "how do these pieces fit
  together at a service-call level."
- This doc — use for "what specs do I need on my blueprint to make the
  entity behave like a vanilla Y."

## How the two pipelines work

Timberborn assembles a prefab from a blueprint by walking the blueprint's
`ComponentSpec` list and asking, for each spec, two questions:

1. **Decorator pipeline (`TemplateModule.AddDecorator<TSpec, TComponent>`).**
   For each `Configurator` in the active context (`Game`, `MapEditor`),
   Bindito reads its registered `TemplateModule` and, for every decorator
   pair `(TSpec → TComponent)`:
   - If the prefab already has a component (or spec) of type `TSpec`,
     attach a new instance of `TComponent` and inject it.
   - This is *recursive in shape*: the LHS of a decorator can be another
     decorator-added component (e.g.
     `AddDecorator<Demolishable, ReachableDemolishable>` fires only after
     `AddDecorator<DemolishableSpec, Demolishable>` has already attached
     `Demolishable`). The build pass iterates until quiescence.
   - Source types can also be **interfaces** (`AddDecorator<ITransmitter, Automator>`)
     — any component implementing the interface triggers the decorator.

2. **Prefab-converter pipeline (`ISpecToPrefabConverter`).**
   A small, fixed set of specs are *not* handled by decorators. They are
   processed at prefab-build time by `MultiBind<ISpecToPrefabConverter>`
   bindings. There are exactly three of these in vanilla:

   | Spec | Converter | Effect |
   |---|---|---|
   | `TransformSpec` | `TransformSpecPrefabConverter` | Sets the child `GameObject`'s local position/rotation/scale. |
   | `CollidersSpec` | `CollidersSpecPrefabConverter` | Adds `BoxCollider` / `SphereCollider` / `CapsuleCollider` Unity components according to the spec's arrays. |
   | `TimbermeshSpec` | `TimbermeshSpecConverter` | Loads a `.timbermesh` asset and attaches a `MeshFilter` + `MeshRenderer`. |

   These are bound in `UnityEngineSpecsConfigurator` and
   `TimbermeshConfigurator`. They run *before* the decorator pass, so by
   the time decorators look at the prefab, meshes, colliders, and child
   transforms are already in place.

The practical consequence: **if a piece of behavior depends on a Unity
component being on the GameObject (a `Collider` for raycast picking, a
`Renderer` for visibility), check the prefab-converter pipeline. If it
depends on a `BaseComponent` being attached (`Demolishable`, `Building`,
`NaturalResource`, `LabeledEntity`), check the decorator pipeline.** Both
must succeed independently for an entity to be fully wired up.

## Foundational seams

These are the specs whose presence on a `BlockObject` blueprint pulls in
the chains everything else depends on. Get any one of them wrong and
many downstream subsystems break silently.

### `TemplateSpec`

The root identity spec — *every* entity that goes through the
`TemplateModule` pipeline carries one. It records the template name
(used cross-faction: `House` template → `House.Folktails` and
`House.IronTeeth` blueprints share a template name), feature-toggle
gating, and back-compat alias names.

Direct decorators: attaches `InstantiatedTemplate` (`TemplateSystemConfigurator`)
and `HighlightableObject` (`SelectionSystemConfigurator`). The latter is
what lets the selection cursor draw the outline halo over the entity.

### `BlockObjectSpec`

The spatial-identity spec — declares the entity occupies a footprint of
voxels in the world grid. Pulls in the entire spatial-aware chain:

`BlockObjectSpec → BlockObject → { TransformController, PlacementChangeNotifier, BlockObjectCenter, BlockObjectRange, BlockObjectState, BlockObjectPostLoadState, Preview, PreviewBlockObject, BlockObjectAtopDeletionBlocker }`
(from `BlockSystemConfigurator`).

`BlockObject` then gets four more chained decorations from other
configurators: `ParentedNeighborCalculator` and `BlockObjectAccessGenerator`
(`BlockObjectAccessesConfigurator`), `BlockObjectNavMesh`
(`BlockSystemNavigationConfigurator`), `BlockObjectGridFootprint`
(`ConstructionGuidelinesConfigurator`), `EntranceMarkerDrawer`,
`SelectableObject`, `BlockObjectDeletionDescriber`
(`BlockSystemUIConfigurator`), `StatusSlotOccupier`
(`StatusSystemConfigurator`), `Understructure`
(`UnderstructureSystemConfigurator`), and
`TerrainPhysicsDeletionBlocker` (`TerrainPhysicsConfigurator`).

**`SelectableObject` is attached via the `BlockObject` decorator chain,
not directly bound to any spec.** This means any entity with
`BlockObjectSpec` ends up "selectable in principle" at the component
level. Whether it is *actually* selectable depends on the prefab-converter
pipeline producing a collider for the cursor raycast to hit — see
`CollidersSpec` below.

**`BlockObjectSpec.Overridable` has a load-bearing side-effect on
`Demolishable`.** When `Overridable: true`, `Demolishable.Mark()`
short-circuits to `EntityService.Delete(this)` — no beaver work, instant
delete. The relevant excerpt:

```csharp
public void Mark()
{
    if (_blockObject.Overridable)
    {
        _entityService.Delete(this);
    }
    else if (!IsMarked)
    {
        IsMarked = true;
        _demolishJob.Enable();
        // ...
    }
}
```

And `_blockObject.OverridableChanged += OnOverridableChanged` is wired so
that an entity which becomes overridable mid-game also gets instant-
deleted. Use `Overridable: false` if you want the demolition to be a
real beaver job; `Overridable: true` is the right choice for nature
elements the player can "remove instantly with the demolish tool"
(saplings, grass, decorative props).

### `BlockObjectModelSpec`

Selects the visual model for the block. Decorates the prefab with
`BlockObjectModel` and, indirectly via the `IBlockObjectModel` interface,
`BlockObjectModelController` (`BlockObjectModelSystemConfigurator`). The
controller drives finished/unfinished/preview switching. Custom flora
that doesn't need that switching can either omit this spec or use it
purely as a model-attachment hook.

### `LabeledEntitySpec`

The "this entity has a display name" spec. Attaches `LabeledEntity`
(`EntitySystemConfigurator`), and via the `LabeledEntity` decorator
chain, also `LabeledEntityNamer` (`EntityNamingConfigurator`). Without
`LabeledEntitySpec` the entity panel can't render a header or
description string.

`PlaceableBlockObjectSpec` also independently attaches `LabeledEntityBadge`
(the 3D world-space label) and `PlaceableBlockObjectDescriber` (the
entity-panel describer). Natural entities that aren't placeable should
use `LabeledEntitySpec` alone; player-placeable buildings get both.

### `BasicSelectionSoundSpec`

Plays a sound when the entity is clicked. Single decorator:
`BasicSelectionSoundSpec → BasicSelectionSound` (`CoreSoundConfigurator`).
`BuildingSpec` already triggers `BuildingSelectionSound` and
`BuildingSounds` (which read `BuildingSpec.SelectionSoundName` and
`LoopingSoundName`), so player-built structures don't need this spec.
Natural elements that aren't `BuildingSpec` but want a click sound do.

### `BoxColliderAdderSpec`

A *decorator-pipeline* way to add a `BoxCollider` to the GameObject —
distinct from `CollidersSpec` (which is prefab-converter pipeline).
Decorator: `BoxColliderAdderSpec → BoxColliderAdder`
(`SelectionSystemConfigurator`). The `BoxColliderAdder` runs in
component-lifecycle (`Awake`/`Start`) and calls
`gameObject.AddComponent<BoxCollider>()` with the spec's center/size
values.

Two reasons this exists alongside `CollidersSpec`:

1. **Per-child-mesh attachment.** `CollidersSpec` operates on the
   GameObject the converter is invoked on (typically a child mesh
   GameObject). `BoxColliderAdderSpec` is read on the root entity,
   producing a collider on the root that doesn't have to match any mesh.
2. **Late attachment.** Because it goes through the decorator pipeline,
   it can be combined with other decorator-pipeline behavior in ways the
   prefab-converter pipeline can't.

For most Keystone-authored entities composing vanilla meshes, the
**right choice is `CollidersSpec` on each child** (so the cursor raycast
hits the actual mesh extent). `BoxColliderAdderSpec` is the right choice
when you want a single hit-box around an entity regardless of internal
mesh layout.

## Natural-resource family

The set of specs that mark an entity as "spawned by the map / not
player-placed" and that drive the moisture/contamination/reproduction
lifecycle.

### `NaturalResourceSpec`

Catch-all marker for spawn-system entities. Drives a fat chain:

`NaturalResourceSpec → { NaturalResource, Demolishable, CoordinatesOffsetter, LivingNaturalResource, DyingNaturalResource }`
(`NaturalResourcesConfigurator`),
plus `Reproducible` (`NaturalResourcesReproductionConfigurator`),
`NaturalResourceModel` + `NaturalResourceCenterProvider`
(`NaturalResourcesModelSystemConfigurator`),
`NaturalResourceEntityBadge` + `NaturalResourceDescriber`
(`NaturalResourcesUIConfigurator`),
`UncuttableReacher` + `UncuttableRemoveYieldStrategy`
(`UncuttableYieldingConfigurator`),
`StatusSubject` in MapEditor context (`MapEditorNaturalResourcesConfigurator`).

Note **`NaturalResourceSpec` attaches `Demolishable` automatically** —
so natural resources don't need to *also* carry `DemolishableSpec` to be
markable for demolition. But `DemolishableSpec` is what carries the
`DemolishTimeInHours` and `ShowDemolishButtonInEntityPanel` configuration,
so if either of those needs to be non-default the spec must still be
present.

The `Reproducible` decoration is what makes natural resources participate
in the natural-spread system; `LivingNaturalResource` and
`DyingNaturalResource` are decorator-only proxy components that other
decorators can hook onto (`DyingNaturalResource → DyingNaturalResourceReproducible`,
`LivingNaturalResource → LivingReproducible`).

### `WateredNaturalResourceSpec`

Flora that needs moisture. Decorators: `WateredNaturalResourceSpec → WateredNaturalResource`,
then `WateredNaturalResource → DryObject` (the runtime component that
reads from `SoilMoistureService`). In MapEditor context it also gets
`InstantNaturalResource` and `DryObject` directly so it skips the
growth lifecycle.

UI decorators: `WateredNaturalResourceSpec → WateredNaturalResourceDescriber`,
`WateredNaturalResource → WateredNaturalResourceStatus`. The status
component is what surfaces the "dying of thirst" badge.

`DaysToDieDry` on the spec is the grace period before `DryObject`
escalates to entity deletion.

### `FloodableNaturalResourceSpec`

Flora with a viable water-depth window. Decorators:
`FloodableNaturalResourceSpec → LivingWaterObject`,
then `LivingWaterObject → LivingWaterNaturalResource`. The
`LivingWaterNaturalResource` component watches the water column depth
and triggers the death timer when out of `[MinWaterHeight, MaxWaterHeight]`
for `DaysToDie`.

UI decorators: `FloodableNaturalResourceSpec → FloodableNaturalResourceDescriber`,
`LivingWaterNaturalResource → LivingWaterNaturalResourceStatus`.

### `ContaminatedNaturalResourceSpec`

Flora that requires *contaminated* soil (Mangrove, badwater-adapted
plants). Decorators:
`ContaminatedNaturalResourceSpec → ContaminatedNaturalResource`,
then `ContaminatedNaturalResource → ContaminatedObject`. Mirrors the
moisture chain but reads from `SoilContaminationService` and treats
contamination as a *requirement* (entity dies when contamination drops
below threshold), not a hazard.

Status: `ContaminatedNaturalResource → ContaminatedNaturalResourceStatus`.

### `GrowableSpec`

The growth-lifecycle spec — anything that goes from planted to mature.
One decorator: `GrowableSpec → Growable`. The `Growable` component
exposes `GrowthProgress` (0..1) and fires the
"finished growing" notification that downstream behavior can listen
to. `GrowthTimeInDays` is the spec's only field.

### `ReproducibleSpec`

Actually not a vanilla spec — reproduction is wired off `NaturalResourceSpec`
via the `Reproducible` decoration (see above). Keystone code that wants
"reproduce like a tree" should compose `NaturalResourceSpec`, not invent
a separate marker.

### `PlantableSpec`

The "foresters/farmers can plant this" spec. Decorator:
`PlantableSpec → Plantable` (`PlantingConfigurator`). Companion
decorator on `Reproducible`: `Reproducible → PlantableReproductionBlocker`
— this *blocks* the natural-reproduction system from spreading
plantable seedlings on their own (the player plants them via a forester
instead). The `ResourceGroup` field on the spec matches against
`PlanterBuildingSpec.PlantableResourceGroup`.

### `CuttableSpec`

The "lumberjack can fell this" spec. Decorators:
`CuttableSpec → Cuttable`, then `Cuttable → GoodStack`
(`CuttingConfigurator`). `GoodStack` holds the carried-good
representation that beavers haul away after the cut. Also indirectly:
`LivingNaturalResource → EmptyDeadNaturalResourceOverrider` makes dead
naturals visually empty.

The `Yielder` nested in `CuttableSpec` is the actual yield record (good
type, amount, removal time). `RemoveOnCut` governs stump persistence,
not yield.

### `GatherableSpec`

The "gatherer can harvest from this" spec. Decorator chain:
`GatherableSpec → Gatherable`, then `Gatherable → { GatherableYieldGrower, GoodStack, GatherableModel }`.
`GatherableYieldGrower` is what regrows fruit/berries/sap between
gatherings; `GatherableModel` is the swap that shows the "ready vs.
empty" appearance.

A blueprint can hold *both* `CuttableSpec` AND `GatherableSpec` — Pine,
Maple, ChestnutTree, Mangrove are all in this category. The discriminator
between fruit-style gather and sap-style tapping is the nested
`Yielder.ResourceGroup` string (`Gatherable` vs `Tappable`).

### `TreeComponentSpec` / `BushSpec`

Markers that identify a blueprint as a tree or bush. Tree decorations:
`TreeComponentSpec → TreeComponent`, then chained: `TreeComponent → { LumberjackGoodStackAdder, TreeRemoveYieldStrategy, TreeReacher }`
(`ForestryConfigurator`), `TreeComponent → TreeShaker`
(`ForestryEffectsConfigurator`). `TreeReacher` is the wiring that lets
beavers find a stand-position to fell from. `BushSpec → LumberjackGoodStackAdder`
is the lone direct decoration on bushes.

### `TreeCuttingRadiusSpec`

Not a decorator-driven spec — it's data consumed by the lumberjack flag
range computation. No `AddDecorator` references; treat it as inert config
on the blueprint.

### `NaturalResourceModelRandomizerSpec`

Visual variety. Decorator:
`NaturalResourceModelRandomizerSpec → NaturalResourceModelRandomizer`
(`NaturalResourcesModelSystemConfigurator`). The randomizer picks one
of several mesh variants on spawn so a forest doesn't look uniform.

### `YielderSpec`

Not a top-level spec — it's nested inside `CuttableSpec.Yielder`,
`GatherableSpec.Yielder`, and similar. Its fields:

```
string         YielderComponentName     // which component owns this yield
GoodAmountSpec Yield                     // good type + amount
float          RemovalTimeInHours
string         ResourceGroup             // "Gatherable" / "Tappable" / etc
```

Decorator-side, `Yielder` itself gets decorated with `Reservable`
(`YieldingConfigurator`) and the `Worker` worker-side gets behavior
components `YielderRemover`, `YieldRemoverBehavior`,
`YieldRemovalSuccessValidator`. The
`InRangeYielders` interface decorator (`IYielderRetriever → InRangeYielders`)
is what connects yielders to flags / buildings with a `BuildingWithTerrainRange`.

## Demolish family

The cluster that lets the player mark something for destruction and
makes a beaver come do it. Diagnosis-critical for any Keystone entity
that the demolish tool ignores.

### `DemolishableSpec`

Core spec. Decorator chain (`DemolishingConfigurator`):

```
DemolishableSpec  → Demolishable
Demolishable      → DemolishablePrioritizableEnabler
Demolishable      → DemolishJob
Demolishable      → ReachableDemolishable
Demolishable      → Reservable
Demolishable      → StatusSubject
DemolishablePrioritizableEnabler → BuilderPrioritizable
Worker            → Demolisher              (added to every Worker entity)
Demolisher        → DemolishBehavior
```

UI: `Demolishable → DemolitionBlockedStatus` (`DemolishingUIConfigurator`).

`Demolishable.Mark()` is the entry point hit when the player selects
the entity with the demolish tool. Three behaviors depending on
`BlockObjectSpec.Overridable`:

- `Overridable: true` → instant delete via `EntityService.Delete`. No
  beaver work, no `DemolishJob`, no status icon.
- `Overridable: false`, not yet marked → set `IsMarked = true`, enable
  `DemolishJob`, fire `DemolishableMarkedEvent`. The job queues the
  entity into `DemolishJobs` and a beaver eventually walks over and
  works it off.
- Already marked → no-op.

The `DemolishTimeInHours` field on the spec is how long the beaver
spends actually demolishing once they arrive; `_demolishTimeLeft` is
clamped and saved.

`ShowDemolishButtonInEntityPanel` toggles whether the entity panel
exposes a per-entity "demolish" button independent of the area-select
demolish tool.

### `DemolishableFromTopSpec`

**The reach-side spec.** This is what makes a beaver actually able to
reach a `Demolishable` entity for a job. Decorators
(`DemolishingConfigurator`):

```
DemolishableFromTopSpec → AccessibleDemolishableReacher
DemolishableFromTopSpec → BlockObjectAccessible
DemolishableFromTopSpec → HighBlockObjectAccessesAdder
```

`BlockObjectAccessible` exposes an `Accessible` (a beaver stand-point);
`AccessibleDemolishableReacher` is the `DemolishableReacher`
implementation that uses it as the navigation destination;
`HighBlockObjectAccessesAdder` adds the "you can stand on top" approach
voxels for tall objects.

`ReachableDemolishable.Start()` reads `GetComponent<BlockObjectAccessible>()` —
**if it's null (no `DemolishableFromTopSpec`), the entity is permanently
treated as unreachable by the demolish job.** The mark itself might
succeed (you'll see the status icon), but no beaver will ever come.

`DemolishableSelectionTool.ProcessInput()` (the area-select demolish
tool) calls `_areaBlockObjectPicker.PickBlockObjects<Demolishable>(...)`,
which filters on the `Demolishable` component existing — *not* on
reach. So the cursor will happily highlight a `DemolishableFromTopSpec`-less
entity; clicking it will call `Mark()`; the entity will sit marked
forever. That's the "marked but never demolished" failure mode.

### `DemolishableScienceRewardSpec`

Awards science points when the entity is demolished. Decorators:
`DemolishableScienceRewardSpec → DemolishableScienceReward`
(`DemolishingConfigurator`),
`DemolishableScienceRewardSpec → DemolishableScienceRewardDescriber`
(`MapEditorDemolishingUIConfigurator`).

Vanilla uses this on Ruins and similar map-feature blockers; ordinary
buildings and natural resources don't carry it.

### `DemolishableParticleControllerSpec`

Plays particles during demolition. Decorator:
`DemolishableParticleControllerSpec → DemolishableParticleController`.

### `DemolishableStatusIconOffsetterSpec`

Adjusts where the "marked for demolition" icon hovers above the entity.
Decorators:
`DemolishableStatusIconOffsetterSpec → DemolishableStatusIconOffsetter`,
then `DemolishableStatusIconOffsetter → MarkerPosition`.

## Building family

The cluster that marks an entity as player-built infrastructure (cost
in goods, construction site, deconstruction, employment).

### `BuildingSpec`

The "this is a player-built thing" spec. Direct decorators
(`BuildingsConfigurator`):

```
BuildingSpec → Building
BuildingSpec → BuildingSelectionSound
BuildingSpec → BuildingSounds
BuildingSpec → BlockableObject
BuildingSpec → PausableBuilding
```

Plus a *long* tail of subsystem decorations:
`BuildingSpec → BuildingStatusIconOffsetter / StatusSubject / StatusIconCycler`
(`BuildingStatusesConfigurator`),
`BuildingSpec → Deconstructible` (`DeconstructionSystemConfigurator`),
`BuildingSpec → BuildingNavMesh / ConstructionSiteAccessible / BlockableEntranceBuilding`
(`BuildingsNavigationConfigurator` + `BuildingsReachabilityConfigurator`),
`BuildingSpec → ConstructionModeModel`
(`ConstructionModeConfigurator`),
`BuildingSpec → AreaBoundsDrawingBlocker / BuildingAreaBoundsDrawingBlocker`
(`BuildingsUIConfigurator`),
`BuildingSpec → RecoverableGoodProvider`
(`RecoverableGoodSystemConfigurator`),
`BuildingSpec → PreviewWaterInputPipeBlocker`
(`WaterBuildingsConfigurator`).

Note `BuildingSpec` is what makes the entity *not* count as a "natural"
in Keystone's classification — see `BlockObjectClassification.IsNatural`.

### `PlaceableBlockObjectSpec`

The "player can place this with the construction tool" spec. Decorators:
`PlaceableBlockObjectSpec → LabeledEntityBadge`,
`PlaceableBlockObjectSpec → PlaceableBlockObjectDescriber`
(`BlockSystemUIConfigurator`). `ToolGroupId` is what bucket of the
construction menu the entity lives in.

### `BuildingModelSpec`, `BuildingModelGroundCutoffSpec`, `BuildingTerrainCutoutSpec`

Visual specs:
`BuildingModelSpec → BuildingModel`,
`BuildingModelGroundCutoffSpec → BuildingModelGroundCutoff`,
`BuildingTerrainCutoutSpec → BuildingTerrainCutout`
(all `BuildingsConfigurator`). The first is the standard model-switcher
(finished/unfinished/preview); the others handle terrain-occlusion
edge cases.

### `BlockObjectNavMeshAdderSpec`

Adds nav-mesh patches to a `BlockObject`'s footprint. Decorator:
`BlockObjectNavMeshAdderSpec → BlockObjectNavMeshAdder`
(`BlockSystemNavigationConfigurator`). Use when an entity sits in the
nav mesh in a way the default `BlockObjectNavMesh` doesn't capture
(stairs, ramps, multi-level footprints).

### `BlockObjectNavMeshSettingsSpec`

**Not** a decorator-driven spec — searches of all 1033 decorator pairs
turn up zero `AddDecorator<BlockObjectNavMeshSettingsSpec, …>` calls.
It's a pure-data spec consumed at navigation-graph build time. Carry it
on the blueprint when you need to tweak the nav-mesh params, but don't
expect a runtime component named after it.

### `EnterableSpec`

The "real building" discriminator from a Keystone classification
standpoint — houses, district center, farmhouse, stockpiles,
workplaces. Decorator: `EnterableSpec → Enterable`. Beavers entering
the building register as `Enterer` (`Character → Enterer`,
`EnterableSystemConfigurator`).

### `WorkplaceSpec`

Employs beavers. Decorators (`WorkSystemConfigurator`):
`WorkplaceSpec → Workplace`, then `Workplace → { NothingToDoInRangeStatus, WorkplacePriority, WorkplaceWorkerType, WorkplaceWorkingHours }`,
plus `Workplace → WorkplaceDescriber` and `Workplace → NoUnemployedStatus`
(`WorkSystemUIConfigurator`). The `Worker` side (per-beaver) gets
attached via `WorkerSpec → Worker`, then `Worker → { WorkRefuser, WorkerWorkingHours, UnreachableWorkplaceUnassigner }`.

## Prefab-converter (`ISpecToPrefabConverter`) pipeline

A separate, far smaller mechanism for specs whose effect is "add a
Unity component" rather than "attach a `BaseComponent`."

### `TransformSpec`

Sets the GameObject's local transform. Used on child GameObjects of a
composite blueprint to position individual meshes. Position values are
in tile units centered on the parent tile (see Keystone's
`feedback_transformspec_pivot.md` memory note).

### `CollidersSpec`

Adds Unity colliders. The converter pulls three immutable arrays from
the spec:

```csharp
ImmutableArray<BoxColliderSpec>     BoxColliders;
ImmutableArray<SphereColliderSpec>  SphereColliders;
ImmutableArray<CapsuleColliderSpec> CapsuleColliders;
```

For each entry it calls `gameObject.AddComponent<T>()` and sets the
center/size/radius. **The converter is invoked on whatever GameObject
holds the spec** — so a `CollidersSpec` nested in a child mesh's
component list produces a collider on that child mesh, not the root.

**Selection failure case:** the standard cursor-pick path
(`BlockObjectRaycaster` / `SelectableObjectRaycaster`) shoots a ray
through the scene and resolves the hit's `SelectableObject` via the
collider's `GameObject` ancestry. **No collider anywhere in the prefab
tree = no raycast hit = no selection**, even though the entity carries
`SelectableObject` as a component (it's there from the `BlockObject`
decorator chain). This is the most common "why isn't my custom entity
clickable" cause.

For a composite blueprint where children carry `TimbermeshSpec`, the
fix is to put a `CollidersSpec` next to each `TimbermeshSpec` on the
same child, sized to roughly cover the mesh. Use box colliders for
trunk-like meshes, spheres for leafy crowns, capsules for vertical
stems.

### `TimbermeshSpec`

Loads a `.timbermesh` asset and attaches mesh-rendering Unity
components. Bound by `TimbermeshConfigurator`. Has its own concerns
(material lookup against `MaterialCollection.*.blueprint.json`,
mesh-asset registration) covered in `docs/timberborn-api.md`.

## Per-symptom diagnostics

| Symptom | Likely missing | Why |
|---|---|---|
| Cursor click does nothing on the entity | `CollidersSpec` on each visible child mesh (or `BoxColliderAdderSpec` on the root) | `SelectableObjectRaycaster` needs *some* collider in the prefab tree for raycast pick to register. |
| Entity panel header is empty / nothing happens on selection | `LabeledEntitySpec`, or for player-built things `BuildingSpec` (which pulls describer chain) | `LabeledEntity` is the prerequisite for `LabeledEntityNamer` / entity-panel describers. |
| No selection sound on click | `BasicSelectionSoundSpec` (for natural entities) or `BuildingSpec` (for built ones — pulls `BuildingSelectionSound`) | Sound bindings hang off these two specs only. |
| Mark-for-destruction tool can't pick up the entity | `DemolishableSpec` *and* `DemolishableFromTopSpec` | `Demolishable` enables the mark; `BlockObjectAccessible` from `DemolishableFromTopSpec` enables the reach check. |
| Entity gets marked (icon shows) but beaver never arrives | `DemolishableFromTopSpec` | `ReachableDemolishable.IsUnreachable()` returns true when `BlockObjectAccessible == null`. |
| Beaver arrives, plays animation, then nothing happens | `DemolishableSpec.DemolishTimeInHours` is zero, or `Worker → Demolisher` decoration missing on the beaver | Worker decoration is automatic for all `WorkerSpec` entities; if missing, only mods that strip beavers should cause it. |
| "Marked" entity instantly disappears with no animation | `BlockObjectSpec.Overridable: true` | `Demolishable.Mark()` short-circuits to `EntityService.Delete` when overridable. Intentional for nature elements; bug for ordinary buildings. |
| Custom flora doesn't dry up / die from drought | `WateredNaturalResourceSpec` missing or `DaysToDieDry` too large | `WateredNaturalResource → DryObject` is the death-timer wiring. |
| Custom flora doesn't die from contamination | `ContaminatedNaturalResourceSpec` (if it should *thrive* in contamination and die without it) | The vanilla "contaminated *required*" semantics — opposite of beaver injury. |
| Custom flora doesn't reproduce | `NaturalResourceSpec` missing (carries the `Reproducible` decorator) or `Reproducible → PlantableReproductionBlocker` is firing (because `PlantableSpec` is present) | `PlantableSpec` *blocks* natural reproduction by design — foresters plant it instead. |
| Lumberjack ignores the tree | `TreeComponentSpec` or `CuttableSpec` missing | `TreeComponent → TreeReacher` is the navigation hook; `Cuttable → GoodStack` is the harvest accumulator. |
| Gatherer ignores the bush | `GatherableSpec` missing, or `Gatherable.Yielder.ResourceGroup` mismatches the gatherer flag's good-allow list | `GatherableYieldGrower` runs continuously once `Gatherable` is attached; the gatherer's filter is the second-stage check. |
| Entity panel describer says "no description" | `LabeledEntitySpec` present but no subsystem-specific describer is attached (e.g. `NaturalResourceDescriber`, `BuildingDescriber`, `Demolishable → DemolitionBlockedStatus`) | Describers are spec-conditional decorations from the relevant UI configurator. |

## Case study: KeystoneRockCluster1

The bug that motivated this doc: `KeystoneRockCluster1.blueprint.json`
composes `TemplateSpec + KeystoneVariantSpec + BlockObjectSpec` (with
`Overridable: false`) + `NaturalResourceSpec + FloodableNaturalResourceSpec + DemolishableSpec + BlockObjectModelSpec`, with `Children`
each carrying `TimbermeshSpec + TransformSpec` (and a tint spec), but
*no* `CollidersSpec` on the children and *no* `DemolishableFromTopSpec`
on the root.

Observed: clicking the rock cluster does nothing (selection fails);
running the demolish tool over the area highlights nothing on hover and
can't pick the cluster up.

Diagnosis using this doc's diagnostic table:

1. **Selection fails** ("cursor click does nothing on the entity"): no
   collider in the prefab tree. `BlockObjectSpec` attached
   `SelectableObject` via the decorator chain, but
   `SelectableObjectRaycaster` needs a Unity collider for its raycast
   to resolve. Each `TimbermeshSpec` child is rendered without an
   accompanying `CollidersSpec`, so the GameObject for each rock has
   `MeshFilter` + `MeshRenderer` but no `Collider`. The root has no
   collider either. The cursor ray passes straight through.

2. **Mark-for-destruction fails** ("mark-for-destruction tool can't
   pick up the entity"): the tool's `AreaBlockObjectPicker.PickBlockObjects<Demolishable>`
   actually *does* find the `Demolishable` component (it's been
   attached by `DemolishableSpec → Demolishable` and by
   `NaturalResourceSpec → Demolishable` — twice over, harmless), but
   the tool's preview-and-action callback goes through the same picker,
   which uses raycast hits. **Same root cause as #1.** And even if the
   raycast found something, the reach check would fail next: no
   `DemolishableFromTopSpec` means no `BlockObjectAccessible`, so
   `ReachableDemolishable.IsUnreachable()` would return true after the
   mark and no beaver would come.

Fix: add `CollidersSpec` to each child mesh's spec list (a single
`BoxColliderSpec` sized to the mesh bounds is sufficient) *and* add a
top-level `DemolishableFromTopSpec` to the root blueprint. With both,
the cursor hits the children, resolves `SelectableObject` on the
parent, and the demolish tool can both mark and dispatch a beaver
demolishing job. `Overridable: false` is correct for a rock cluster
(you want the demolition to be a real beaver job, not an instant
delete) and should stay.

The general lesson — both pipelines are independent. Selection takes a
Unity-component-side configuration (`Collider`) *and* a `BaseComponent`-
side configuration (`SelectableObject`); demolish takes a mark-side
configuration (`Demolishable`, attached automatically via two paths)
*and* a reach-side configuration (`BlockObjectAccessible` via
`DemolishableFromTopSpec`). Forgetting either side leaves the feature
broken in a way that doesn't throw an exception — it just silently
doesn't work.

### Outcome (2026-05-12 — partial)

The fixes above resolved **selection** as predicted: adding
`CollidersSpec` to each child mesh made the cursor raycast register,
and adding `LabeledEntitySpec` (with placeholder loc keys + a vanilla
icon path) stopped `EntityPanel.UpdateEntityBadge` from null-derefing
on the empty label. Class C entities now select cleanly.

**Mark-for-destruction stayed broken** even after every measure the
doc's diagnostics table suggested:

- `DemolishableSpec` was present from the start (auto-attaches
  `Demolishable + DemolishJob + Reservable + BuilderPrioritizable + ReachableDemolishable + StatusSubject`).
- `DemolishableFromTopSpec` added (auto-attaches
  `AccessibleDemolishableReacher + BlockObjectAccessible + HighBlockObjectAccessesAdder`).
- `CollidersSpec` on every child mesh.
- `KeystoneVariant.Class = "C"` stamped at spawn (so
  `AmbientNaming.IsAmbient()` returns false and our
  `DemolishableSelectionToolPatch` doesn't filter the entity).
- After spawn, the entity was pushed through the full
  `EntityComponent.PreInitialize → Initialize → PostInitialize → PostLoad`
  lifecycle (the "run it through the initializer" hint from
  Discord), since `BlockObjectFactory.CreateFinished` only calls
  `BlockObject.MarkAsFinished()` and the `IFinishedStateListener`
  notification chain is gated on `BlockObjectState._initialized`,
  which only flips during `PostInitializeEntity`.

A runtime diagnostic dump of `BaseComponent.AllComponents` confirmed
that all expected runtime components were attached on the entity (81
of them, including all the demolish-family components above). Selection
worked. Hover raycast worked. The `DemolishableSelectionTool`'s patch
fired on its first invocation (confirming the tool was active and our
patch resolved), and `AmbientNaming.IsAmbient` correctly returned false
for the Class-C cluster — so our patch *passed* the entity through.

Yet the tool's drag-area picker never put the cluster in the
filter-input list. The cluster was both rendered + collidable (cursor
hovers/clicks register) and **could be stacked on by another cluster
in the dev tool**, indicating `IBlockService` *also* didn't see it
post-`AddToService` even though `Reposition` inside `CreateUnfinished`
calls `AddToService` directly. The two observations together imply
there's a third pipeline beyond decorator-attached components and
prefab-converter-attached colliders — likely some service-registration
step that fires off `EntityInitializedEvent` or a state-listener — that
isn't running for entities spawned via the dev tool's path, even with
the manual init push.

Open question to investigate when this work resumes: **what specific
service registration is missing for the dev-tool spawn path that vanilla
in-game placement gets for free?** Candidate angles:

- Look at vanilla building/construction-site code paths that finalize a
  placed entity. The instant `BlockObjectFactory.CreateFinished` route
  used by both `NaturalResourceFactory` and the dev tool likely isn't
  the production placement path — the real path probably goes through
  `ConstructionSite`/`Builder` for buildings, or some other facade for
  natural-resource initial-map population.
- Inspect `BlockObjectFinishedStateListener` (and similar) — anything
  named like `*FinishedStateListener` that calls into a service may be
  the listener whose registration we're missing.
- Confirm whether the `EntityInitializedEvent` we fire by calling
  `EntityComponent.Initialize` is reaching all the service listeners
  it should. Add a temporary `EventBus.Register` log to verify.
- Cross-check against the Mechanistry SDK's HelloWorld / ShantySpeaker
  sample mods to see whether they spawn natural-resource-shaped
  entities at runtime in a different way.

Until then, rock clusters ship as Class B (yielding + instant-delete via
`BuildingDeconstructionTool` widened by `BuildingDeconstructionClassBPatch`),
matching the mini-flourish family. The Class C aspiration ("blocking
+ markable for beaver demolition") is parked.

## Stale risk and regeneration

This doc is generated from a Timberborn 1.0 snapshot. The procedure to
regenerate after a game update:

1. Decompile every `*Configurator.cs` in `Timberborn.*.dll` to a working
   directory (vanilla DLL path:
   `<Timberborn install>\Timberborn_Data\Managed\Timberborn.*.dll`).
2. Grep `builder.AddDecorator<` across the decompile output; the raw
   lines are the source for the master table.
3. Re-spot-check the foundational seams (`BlockObjectSpec`,
   `Demolishable.Mark()` body, `CollidersSpecPrefabConverter`) — those
   are the highest-risk surfaces for silent behavior change.

The case study and the per-symptom diagnostics are the prose value here;
the master table is mechanical and regeneration is cheap.
