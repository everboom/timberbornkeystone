# Timberborn 1.0 API reference (curated, services + patterns)

Hand-curated notes and Keystone-usage annotations for Timberborn's DLLs.
Scope: orientation, lifecycle hooks, cross-cutting API patterns, per-service
gotchas. **For `ComponentSpec` types** (BuildingSpec, EnterableSpec, PathSpec,
WateredNaturalResourceSpec, etc. — the prefab-level metadata that drives
ecology classification) see the curated **`timberborn-specs.md`**.

The complete autogenerated reference covering all 490 Timberborn DLLs lives
in **`timberborn-api-full.md`** and is regenerated by
`tools/generate-api-cache.ps1`.

**When to read which:**
- *Looking for a type or "does X exist"?* → `timberborn-api-full.md` (complete index, full signatures).
- *Which spec do I check to discriminate buildings vs. naturals?* → `timberborn-specs.md` (ecology-grouped, with discriminator cheat sheet).
- *How does Keystone use a service, what are the gotchas?* → this file (annotations, two-axis API patterns, lazy-init traps, lifecycle ordering).

This file is hand-edited and intentionally narrow. Add notes here when you discover a non-obvious behavior, a useful pattern, or a way the game's API surprised you. Don't duplicate what the autogenerated dump already shows.

**Stale risk:** regenerate the full reference after Timberborn version bumps; this file's notes are version-resilient unless the underlying API changes.

## SDK sample-mod patterns (from `<timberborn-modding-sdk>`)

The Modding SDK ships four sample mods: **HelloWorld** (UI fragment +
bottom-bar button), **ShantySpeaker** (custom decoration building +
sound), **OverwritesExample** (data-only blueprint overrides),
**YearOfTheSnakeBeaverTails** (asset-only beaver tail decal). Patterns
worth knowing before writing mod code:

### Mod entry points

Two distinct entry points fire at different phases:

- `IModStarter.StartMod(IModEnvironment)` — runs *before* singletons
  are constructed. Use for one-time setup that must happen at process
  start (logging, env-var setup). Does not inject DI services.
  ```csharp
  internal class HelloWorldLogger : IModStarter {
    public void StartMod(IModEnvironment modEnvironment) {
      Debug.Log("Hello World, ...");
    }
  }
  ```
- `[Context("Game")] Configurator.Configure()` — runs once when the
  Game scope activates (i.e. settlement loaded). Register your bindings
  here. Singletons get instantiated immediately after.

There is no `Main()`. `IModStarter` and `Configurator` are how the
runtime finds your code. **Both are class-level** — Bindito and the
mod loader scan all DLLs in the mod folder for either.

### `[Context("Game")]` Configurator structure

Every mod with code has at least one Configurator. Bindings inside:

```csharp
[Context("Game")]
public class MyConfigurator : Configurator {
  protected override void Configure() {
    Bind<MyService>().AsSingleton();                          // self-binding
    Bind<IMyPort>().To<MyAdapter>().AsSingleton();            // interface → impl
    MultiBind<BottomBarModule>().ToProvider<MyProvider>().AsSingleton();
  }
}
```

`MultiBind<T>` is the pattern for collections — UI module providers,
template-module decorators, dev modules, anything where multiple
contributors compose into one list.

### Spec + Component pattern (entity decoration)

Adding behaviour to an entity blueprint:

1. Define a `ComponentSpec` record with `[Serialize]` properties
   (configuration data, written into the blueprint JSON):
   ```csharp
   internal record FinishableBuildingSoundPlayerSpec : ComponentSpec {
     [Serialize] public string SoundName { get; init; }
   }
   ```
2. Define a `BaseComponent` subclass that implements behaviour and
   reads its spec in `Awake()`:
   ```csharp
   internal class FinishableBuildingSoundPlayer : BaseComponent,
                                                  IAwakableComponent,
                                                  IFinishedStateListener {
     [Inject]
     public void InjectDependencies(ISoundSystem soundSystem) {
       _soundSystem = soundSystem;
     }
     public void Awake() { _spec = GetComponent<FinishableBuildingSoundPlayerSpec>(); }
     public void OnEnterFinishedState() { /* ... */ }
   }
   ```
3. Register via `TemplateModule.AddDecorator<TSpec, TComponent>()`:
   ```csharp
   MultiBind<TemplateModule>().ToProvider(() => {
     var b = new TemplateModule.Builder();
     b.AddDecorator<FinishableBuildingSoundPlayerSpec, FinishableBuildingSoundPlayer>();
     return b.Build();
   }).AsSingleton();
   ```
4. Reference the spec by class name in the blueprint JSON:
   ```json
   "FinishableBuildingSoundPlayerSpec": { "SoundName": "Music_..." }
   ```

The `[Serialize]` on the spec property is what writes/reads the value
through the blueprint JSON. **This is the only persistence path
demonstrated in the SDK samples** — see "Persistence" below for what
the samples *don't* show.

### Lists of records on a spec: `ImmutableArray<T>`, not `List<T>`

When a `[Serialize]` spec property holds a list of custom records, the
type **must** be `ImmutableArray<T>`, not `List<T>`. A list-typed
property compiles fine but blows up at game load time:

```
ArgumentException: Can't deserialize System.Object[] to type
System.Collections.Generic.List`1[YourMod.YourRecordEntry]
   at Timberborn.SerializationSystem.PrimitiveTypeSerialization.Deserialize
   at Timberborn.BlueprintSystem.BasicDeserializer.DeserializeSingle
```

The blueprint deserializer's `BasicDeserializer` produces `Object[]` for
JSON arrays and asks `PrimitiveTypeSerialization.Deserialize` to coerce
to the declared property type. `PrimitiveTypeSerialization` only
handles primitives + a handful of registered conversions; `List<T>` for
custom `T` isn't one of them. `ImmutableArray<T>` *is* — vanilla
`BlockObjectSpec.Blocks` is `ImmutableArray<BlockSpec>`, the working
list-of-records pattern in the game's own specs.

Fix shape:

```csharp
public record MyBookSpec : ComponentSpec {
  [Serialize] public ImmutableArray<MyEntry> Entries { get; init; } =
      ImmutableArray<MyEntry>.Empty;
}
```

Tests don't catch this — they exercise the C# types directly without
going through `ISpecService`. The bug only surfaces when the blueprint
is actually deployed to the mod folder and a game session loads it.
Keystone hit this with `KeystoneRecipeBookSpec.Recipes` and
`KeystoneBiomeLevelsSpec.Levels` after the recipe-book content first
shipped to the deployed mod folder.

Consumer-side note: `ImmutableArray<T>` exposes `.Length`, not
`.Count` (the latter is an extension on `IReadOnlyCollection<T>` that
ImmutableArray doesn't pick up by default). Iteration via `foreach`
works the same.

### Component DI: `[Inject]` methods, not constructors

`BaseComponent` subclasses are spawned by Timberborn (not Bindito), so
constructor injection doesn't work. Use a public method decorated with
`[Inject]`:

```csharp
[Inject]
public void InjectDependencies(ISoundSystem soundSystem) {
  _soundSystem = soundSystem;
}
```

Bindito invokes the `[Inject]` method after the component is
instantiated. **Singletons still use constructor injection** — only
components need `[Inject]`.

### Component lifecycle hooks (cheat sheet)

| Interface                 | Method                                | Fires when                          |
|---------------------------|---------------------------------------|-------------------------------------|
| `IAwakableComponent`      | `Awake()`                             | Just after instantiation            |
| `IInitializableEntity`    | `InitializeEntity()`                  | After all components are awakened   |
| `IFinishedStateListener`  | `OnEnterFinishedState/OnExitFinishedState` | Construction completes / entity unbuilt |
| `IDeletableEntity`        | `DeleteEntity()`                      | Entity removed                       |
| `IPersistentEntity`       | `Save(IEntitySaver)/Load(IEntityLoader)` | Save / load (see "Persistence" below) |

Prefer these over manual EventBus subscription when a well-defined
lifecycle event exists — `IFinishedStateListener` is cleaner than
listening for `EntityFinishedConstructionEvent`.

### UI fragment patterns

- **Bottom-bar button** (HelloWorld): `MultiBind<BottomBarModule>` with
  a provider that yields a `BottomBarModule` containing one or more
  elements. UI loaded from `.uxml`/`.uss` via `VisualElementLoader`.
- **Entity panel fragment** (HelloWorld): implement
  `IEntityPanelFragment` with `InitializeFragment()` (build UI once),
  `ShowFragment(BaseComponent)` (decide visibility per selected
  entity, typically by probing `GetComponent<T>()`), `ClearFragment()`,
  `UpdateFragment()` (per-frame refresh while visible).
  ```csharp
  public void ShowFragment(BaseComponent entity) {
    if (entity.GetComponent<NaturalResource>() is not null) {
      _root.ToggleDisplayStyle(true);
    }
  }
  ```

### Persistence (singleton / entity save/load)

**The SDK samples don't demonstrate save/load** — they get persistence
"for free" via `[Serialize]` on `ComponentSpec` properties (which
writes through the blueprint JSON), which is design-time configuration,
not runtime state. For runtime state that must persist across saves
(`IRegionState` values, ecology fields, anything compounding), the
patterns are reverse-engineered from `docs/timberborn-api-full.md` and
reflection over the actual game DLLs.

#### Save key types (all simple `(string)` ctors)

```
Timberborn.WorldPersistence.SingletonKey   ctor(string name); .Name
Timberborn.WorldPersistence.ComponentKey   ctor(string name); .Name
Timberborn.Persistence.PropertyKey<T>      ctor(string name); .Name
Timberborn.Persistence.ListKey<T>          ctor(string name); .Name
```

#### Singleton save/load idiom

A persistent singleton implements **both** `ILoadableSingleton` and
`ISaveableSingleton`, and injects `ISingletonLoader` via its
constructor:

```csharp
public class FooService : ILoadableSingleton, ISaveableSingleton {
  private static readonly SingletonKey Key = new("MyMod_Foo");
  private static readonly PropertyKey<int> CounterKey = new("Counter");
  private static readonly ListKey<int> ItemsKey = new("Items");

  private readonly ISingletonLoader _singletonLoader;
  private int _counter;
  private List<int> _items = new();

  public FooService(ISingletonLoader singletonLoader) {
    _singletonLoader = singletonLoader;
  }

  public void Save(ISingletonSaver saver) {
    var obj = saver.GetSingleton(Key);
    obj.Set(CounterKey, _counter);
    obj.Set(ItemsKey, _items);
  }

  public void Load() {
    // Load() fires even on fresh games (no save). TryGet returns false
    // when nothing is saved -- treat that as "fresh state, nothing to
    // restore."
    if (!_singletonLoader.TryGetSingleton(Key, out var obj)) return;
    _counter = obj.Get(CounterKey);
    _items = obj.Get(ItemsKey);
  }
}
```

Confirmed by reflection on real implementers in
`Timberborn.Bots.BotPopulation`, `Timberborn.WindSystem.WindService`,
`Timberborn.WeatherSystem.TemperateWeatherDurationService` — all
implement the same shape (`ctor(... ISingletonLoader ...)`,
`Save(ISingletonSaver)`, `Load()`).

#### Entity save/load idiom

Components on entities use `IPersistentEntity` instead:

```csharp
Void Save(IEntitySaver entitySaver)
Void Load(IEntityLoader entityLoader)
```

with `entitySaver.GetComponent(componentKey)` returning the same
`IObjectSaver` API as singletons. The `(ComponentKey, string)`
overloads provide a namespaced scope when multiple mods or game
features write to the same component.

End-to-end shape (from Bobingabout `CharacterTextureSetter`):

```csharp
public class MyComponent : BaseComponent, IPersistentEntity {
  private static readonly ComponentKey ComponentKey = new("MyComponent");
  private static readonly PropertyKey<string> ValueKey = new("Value");

  public void Save(IEntitySaver saver) {
    if (!string.IsNullOrEmpty(_value)) {
      saver.GetComponent(ComponentKey).Set(ValueKey, _value);
    }
  }

  public void Load(IEntityLoader loader) {
    if (loader.HasComponent(ComponentKey)) {
      var obj = loader.GetComponent(ComponentKey);
      if (obj.Has<string>(ValueKey)) _value = obj.Get(ValueKey);
    }
  }
}
```

`IObjectLoader` exposes `HasComponent(ComponentKey)` to test existence
before reading and `Has<T>(PropertyKey<T>)` to test individual
properties — avoids exceptions when fields are missing in older saves.

#### Save migration (legacy-key fallback)

When you change save format or rename a key, keep the old `ComponentKey`
around and check it first. `CharacterTextureSetter` does this verbatim:

```csharp
private static readonly ComponentKey CurrentKey = new("CharacterTextureSetter");
private static readonly ComponentKey LegacyKey = new("EmberpeltsCharacterTextureSetter");

public void Load(IEntityLoader loader) {
  if (!LegacyLoad(loader) && loader.HasComponent(CurrentKey)) {
    // ... read from current key
  }
}

private bool LegacyLoad(IEntityLoader loader) {
  if (!loader.HasComponent(LegacyKey)) return false;
  // ... read old shape, translate to new fields
  return true;
}
```

No version field needed — presence of the legacy key is itself the
signal. New saves only write the current key, so loaded-and-resaved
entities transparently migrate forward.

#### Native types you can save without a serializer

`int`, `float`, `bool`, `string`, `char`, `Vector2`, `Vector2Int`,
`Vector3`, `Vector3Int`, `Quaternion`, `Color`, `Guid`. Lists of any of
those via `ListKey<T>`. **No native dictionary support** — flatten to
parallel lists (one of keys, one of values) and zip on load.

#### Custom value types — `IValueSerializer<T>`

For your own structs / records (`RegionId(int)`, `SurfaceCoord`, etc.),
implement:

```csharp
public interface IValueSerializer<T> {
  void Serialize(T value, IValueSaver valueSaver);
  Obsoletable<T> Deserialize(IValueLoader valueLoader);
}
```

`Obsoletable<T>` is a wrapper (`Value`, `Obsolete : bool`) that lets a
deserializer signal "I read this from a deprecated format and upgraded
it." Game examples: `SerializedGoodValueSerializer`,
`SerializedNeedValueSerializer`. Pass the serializer to the
`Set<T>(PropertyKey<T>, T, IValueSerializer<T>)` /
`Get<T>(PropertyKey<T>, IValueSerializer<T>)` overloads, or to
`Set<T>(ListKey<T>, IReadOnlyCollection<T>, IValueSerializer<T>)` /
`Get<T>(ListKey<T>, IValueSerializer<T>)` for collections.

#### Load order: terrain vs. our singletons

All `ILoadableSingleton.Load()` calls fire before any
`IPostLoadableSingleton.PostLoad()`. The order between two `Load()`
calls in the same phase is **not** specified — `TerrainMap.Load()`
might fire before or after our `RegionService.Load()`.

Practical consequence for Keystone: if our singleton needs terrain
geometry (re-Index from terrain on load), do it in `PostLoad()`, not
`Load()`. The clean split is:

- `Load()` — read saved compound state from `ISingletonLoader` into a
  temporary in-memory buffer. Don't touch terrain.
- `PostLoad()` — terrain is now guaranteed loaded; run `Index()`
  from terrain to rebuild geometry, then drain the buffer onto the
  fresh regions and discard the buffer.

#### Mod-collision scoping

Both `ISingletonSaver.GetSingleton` and `IEntitySaver.GetComponent`
have `(key, string)` overloads where the string is a namespace. Use a
mod-unique prefix (`"Keystone.RegionService"`) on save keys to avoid
colliding with the base game or other mods that pick the same name.

### Custom JSON specifications (`ISpecificationService`)

Distinct from the spec-on-blueprint pattern documented above (which is
the game's built-in spec system tied to `Blueprint.Specs`). This is the
mechanism for shipping **arbitrary JSON files** that your mod's code
discovers and reads at load time — useful for per-biome rules, per-mod
content tables, anything authored as data rather than code.

File-naming convention:
`{SpecificationTypeName}.{UniqueId}.json` placed under the mod's
`Specifications/` folder. The game finds them by type name; the
`UniqueId` segment lets multiple mods (or multiple categories from the
same mod) contribute entries without name collisions.

Three pieces:

```csharp
// 1. Plain data class (no base class needed).
public class MyRuleSpecification {
  public string FactionId { get; set; }
  public ImmutableArray<string> AllowedFlora { get; set; }
}

// 2. Deserializer: IObjectSerializer<T>. Save throws -- specs are read-only.
public class MyRuleSpecificationDeserializer : IObjectSerializer<MyRuleSpecification> {
  public void Serialize(MyRuleSpecification v, IObjectSaver s) => throw new NotSupportedException();

  public Obsoletable<MyRuleSpecification> Deserialize(IObjectLoader loader) {
    return new MyRuleSpecification {
      FactionId = loader.Get(new PropertyKey<string>("FactionID")),
      AllowedFlora = loader.Get(new ListKey<string>("AllowedFlora"), /* string serializer */),
    };
  }
}

// 3. Service: ILoadableSingleton that calls ISpecificationService.GetSpecifications<T>.
public class MyRuleSpecificationService : ILoadableSingleton {
  private readonly ISpecificationService _specs;
  private readonly MyRuleSpecificationDeserializer _deserializer;
  private ImmutableArray<MyRuleSpecification> _all;

  public void Load() {
    _all = _specs.GetSpecifications<MyRuleSpecification>(_deserializer).ToImmutableArray();
    // build any indices you need over _all here
  }
}
```

Bind the deserializer + service as singletons in your `Configurator`:

```csharp
containerDefinition.Bind<MyRuleSpecificationDeserializer>().AsSingleton();
containerDefinition.Bind<MyRuleSpecificationService>().AsSingleton();
```

Nested types use child deserializers — the parent's `Deserialize` calls
`loader.Get<TChild>(new ListKey<TChild>("FieldName"), childDeserializer)`
to read a list of nested objects with their own deserializer. See
Bobingabout's `BeaverGrowUpTextureMapSpecification` ↔
`BeaverGrowUpTextureMap` pair for a concrete two-level example.

**When Keystone would reach for this:** any time we want to ship
configurable behaviour as JSON (per-biome thresholds, per-faction
ecology overrides, named ecology rules) instead of compiled code.
Players can drop additional spec files into the mod folder without
recompiling. Cheaper than building a full mod-settings UI for one-shot
overrides; complementary to it for ongoing tuning.

### Data-only mods (OverwritesExample, YearOfTheSnakeBeaverTails)

Pure-data mods don't need C# code at all. `OverwritesExample`
overrides vanilla blueprints by placing `.blueprint.json` files at the
same path as the originals; the game loader prefers mod content over
vanilla. `YearOfTheSnakeBeaverTails` adds new asset bundles
(textures, materials) without code. The mod manifest's
`OptionalMods: [...]` field can declare soft dependencies — the mod
loads regardless, but its content can reference types from the
optional mod when present.

### Tips that bit the sample readers

- **`Player.log` is the diagnostic surface.** `Debug.Log` from any mod
  code goes to `%USERPROFILE%\AppData\LocalLow\Mechanistry\Timberborn\Player.log`.
  No console window. Plan your diagnostics accordingly.
- **Asset paths are bundle-relative, not filesystem-relative.** UI
  assets passed to `VisualElementLoader.LoadVisualElement("HelloWorld")`
  resolve through the mod's asset bundle, not the mod folder.
- **Spec class name = JSON key.** A `FooBarSpec` class is referenced as
  `"FooBarSpec": { ... }` in the blueprint JSON. Renaming the class
  silently breaks loading of existing blueprints.
- **`[Inject]` runs after `Awake()`** but the order between `Awake` and
  `Inject` for sibling components is not specified beyond that — don't
  assume sibling state in `Awake` and don't reach for injected
  dependencies before `Inject` has fired.

## Bindito.Core

Mechanistry's bespoke DI container. The `Configurator` base class exposes
`Bind<T>()` and `MultiBind<T>()` inside `Configure()`. Bindings flow
through three small interfaces (binding builder → scope → export).

### Public API surface

```
// Configurator (Configure() override is called once per Context activation):
ISingleBindingBuilder<T> Bind<T>()
IMultiBindingBuilder<T>  MultiBind<T>()
void Install(IConfigurator)
void InstallAll(string)

// IBindingBuilder<TBound> — pick the implementation:
IScopeAssignee  To<TImplementation>()
IScopeAssignee  ToProvider<TProvider>()
IScopeAssignee  ToProvider(IProvider<TBound>)
IScopeAssignee  ToProvider(Func<TBound>)
IExportAssignee ToInstance(TBound)
IExportAssignee ToExisting<TExisting>()    // alias an existing binding

// IScopeAssignee — pick the lifetime:
IExportAssignee AsSingleton()
IExportAssignee AsTransient()

// IExportAssignee — optional cross-context export:
void AsExported()
```

`ISingleBindingBuilder<T>` and `IMultiBindingBuilder<T>` extend
`IBindingBuilder<T>`; `Bind<T>()` resolves to a single binding while
`MultiBind<T>()` accumulates contributions to a collection (the pattern
behind `MultiBind<TemplateModule>().ToProvider(...)` for entity
decorators).

### Scopes

Use `[Context("Game")]` on the `Configurator` to register inside the
in-game settlement scope. Other contexts exist (menu, etc.); inspect the
SDK reference (`<timberborn-modding-sdk>`) for the exhaustive
list before assuming.

### Common patterns

```csharp
[Context("Game")]
public class MyConfigurator : Configurator {
  protected override void Configure() {
    // Self-binding — resolve MyService anywhere it's injected.
    Bind<MyService>().AsSingleton();

    // Interface → implementation (Keystone uses this for ports/adapters).
    Bind<IMyPort>().To<MyAdapter>().AsSingleton();

    // Provider — when construction needs custom logic.
    MultiBind<TemplateModule>().ToProvider(BuildTemplateModule).AsSingleton();
  }
}
```

The container scans the constructor of every bound type and resolves
each parameter from its other bindings — there is no
`[Inject]`-on-property or service-locator API to reach for. Prefer
constructor injection always.

**Two-axis API.** `SoilMoisture(int)` returns the column's moisture
value (one number per (x,y)). The float is **not** a 0..1 fraction —
it's an "irrigation distance" in tile units, typically 0..~16,
representing how far moisture has propagated inland from the nearest
water source. Treat it as raw scalar tile-units, not a percentage.

surfaces in the same column can return different answers because "is
the soil at this Z moist enough to support life" depends on Z relative
to the water table, not just the raw column value. Keystone surveys
use both: the float feeds the chunked ecology field, the predicate is
recorded on `SurfaceSurvey.IsMoist`.

## Terrain ground recoloring (`TerrainMaterialMap`)

How the ground itself gets recolored — contamination plumes, the
dry/desert vs. irrigated look, **and the "marked for planting" tilled-soil
tint** — is **not** decals or overlay meshes. It's per-tile data channels
baked into the terrain shader (`Shader Graphs/TerrainURP`), all owned by one
singleton: `Timberborn.TerrainSystemRendering.TerrainMaterialMap`.

### The three channels

Each is a `Texture2DArray` in `RG16`, **one slice per Z-level** (fully 3D —
works on overhangs and stacked terrain), bound as a **global** shader texture
via `Shader.SetGlobalTexture`, so every terrain material samples the same maps:

| Shader global | Written by | Drives |
|---|---|---|
| `_DesertIntensityMap` | `SoilMoistureService.SetDesertIntensity` | dry ↔ irrigated; shader blends `_DryFieldTex` / `_WetFieldTex` / `_DesertTex` |
| `_ContaminationMap` | `SoilContaminationService.SetSoilContamination` | badwater tint (`_Contamination_Color`, scale, noise, softness) |
| `_CutoutAndFieldMap` | `ITerrainService.FieldOrCutoutChanged` | **R = is-field (tilled-soil look)**, G = is-cutout |

Dry/irrigated mapping (`SoilMoistureService.UpdateDesertIntensity`):
`moisture == 0` → desertIntensity `1` (full desert); `0 < moisture ≤ threshold`
→ `(1 − moisture/threshold) × maxDesertIntensity`; `> threshold` → `0` (lush).
Recomputed on every moisture change and at load.

### The "marked for planting" tint is the field flag

`PlantingService.SetPlantingCoordinates` → `ITerrainService.SetField(coords)`
flips `_CutoutAndFieldMap.R` to 1 → the terrain shader blends in the tilled
field albedo. `UnsetPlantingCoordinates` → `UnsetField` reverts it; both
crop and tree planting marks go through this, and it's re-applied for every
marked coordinate on load. This is the **persistent** marker (survives closing
the tool, saved with the terrain) — distinct from the transient drag-time
`AreaHighlightingService.DrawTile` overlay quads and from the
`PlantablePreview` grayscale ghost (which `PlantingModeService` hides whenever
the planting tool is closed).

### Update mechanics

Setters are **threshold-gated** (`ChangeThreshold ≈ 1/255`) and enqueue a
change rather than writing immediately. Changes drain on `Tick()` /
`LateUpdateSingleton()`, write into a small buffer `Texture2D`, then
`Graphics.CopyTexture` into **only the dirty Z-slices** (a change on level 3
re-uploads one slice, not the map). `PixelData` is **RG16 = (old, new)**: the
texture carries both values so the shader can **fade the transition over a
tick** (contamination/dryness ease in rather than pop).

### Modding levers (and the alpha trap)

Textures and blend params come from one **singleton** spec,
`TerrainMaterialMapSpec` (`ISpecService.GetSingleSpec`), so they're
**overridable by a blueprint overlay — data-only, no Harmony**: `DesertTexture`,
`DryFieldTexture`, `WetFieldTexture`, `BlendingNoise`, `BlendingNoiseScale`,
`BlendingNoiseMultiplier`, `BlendingSoftness`, `BlendingMargin`,
`AltitudeCeiling`, `AltitudeMultiplier`, `DesertAltitudeMultiplier`,
`CutoutMargin`.

- **"Give the field texture an alpha to fade it" does NOT work.** The terrain
  shader is opaque; the field albedo is a **masked lerp into the base ground**,
  with the blend weight coming from the field map (binary R) modulated by the
  blend noise/margin — *not* from the texture's alpha channel. Alpha in the
  field texture isn't read as a blend weight.
- **Real lever:** swap `WetFieldTexture` / `DryFieldTexture` for a subtler
  asset (closer to base ground) via the spec override. To tune prominence
  while keeping fields legible you author a custom texture (Unity SDK); the
  spec override itself is data-only.
- **Zero-asset experiment:** tweak `BlendingSoftness` / `BlendingMargin` /
  `BlendingNoiseMultiplier` to feather edges — but these are **global** and
  shared with the dry/wet moisture blend, so they also change the moisture
  look. Test and observe.
- **All knobs are global / single-spec:** one texture for *all* fields (tree
  planting marks **and** crop fields). There is no per-tile or per-purpose
  alpha. To soften only Keystone's own marks you'd intercept `SetField`
  (skip it, or Harmony) rather than touch the texture — but check
  `CellIsField` consumers first, since the field flag is terrain state other
  systems may read.

### The field R is a coverage threshold, not opacity (measured)

Tested in-game by writing a fractional R (`0.5`) into the field channel: the
tilled tint did **not** fade uniformly — it broke into **blotches**. So the
terrain shader uses R as a **threshold against `_BlendingNoise`** (field shows
where `noise < R`), *not* as a linear `lerp` weight. Lowering R reduces
*coverage* (dithered patches), never *opacity*. There is therefore **no
data-side route to a uniform alpha**: the only knob that yields a uniform,
less-prominent field at full coverage is the field **albedo texture** itself
(swap `_WetFieldTex` / `_DryFieldTex` for something closer to base ground via
the `TerrainMaterialMapSpec` overlay). A genuine uniform-alpha blend would
require editing the shader's fragment logic — but the terrain shader is
**metadata-only in the asset dump** (`Shader Graphs_TerrainURP.json` has empty
`m_SubPrograms`; no recoverable HLSL/bytecode) and the `.shadergraph` source is
unpublished, so that path is a from-scratch shader reimplementation, not an edit.

## Faction-keyed content (cross-faction findings)

Game content is partitioned per-faction at *load time*, not just at the UI
gate. Four parallel collection systems all use the same Bindito multi-bind
shape — implementations of an `IXxxIdsProvider` interface tell the
corresponding service which collections to load:

| System | Provider interface | Service | What's faction-keyed |
|---|---|---|---|
| Templates | `ITemplateCollectionIdProvider` | `TemplateCollectionService` | Blueprints (buildings, natural resources, characters) |
| Materials | `IMaterialCollectionIdsProvider` | `MaterialRepository` | Meshes + materials referenced by prefabs |
| Goods | `IGoodCollectionIdsProvider` | `GoodService` | Hauled/produced/consumed resource specs |
| Needs | `INeedCollectionIdsProvider` | (need service) | Beaver consumption needs |

The full content set per faction is enumerated by `FactionSpec` —
`TemplateCollectionIds`, `MaterialCollectionIds`, `NeedCollectionIds`,
`GoodCollectionIds`. Plus direct `AssetRef`s for `Textures`,
`PathMaterial`, `BaseWoodMaterial` etc. that aren't collection-driven
and therefore can't be reached via a multi-bind provider.

### Cross-faction unlock pattern

To make the active faction's `TemplateNameMapper` resolve names from
*another* faction (e.g. spawn a Mangrove on Folktails):

```csharp
[Context("Game")]
public class MyConfigurator : Configurator {
  protected override void Configure() {
    MultiBind<ITemplateCollectionIdProvider>()
        .To<MyCrossFactionTemplateProvider>().AsSingleton();
    MultiBind<IMaterialCollectionIdsProvider>()
        .To<MyCrossFactionMaterialProvider>().AsSingleton();
  }
}

public class MyCrossFactionTemplateProvider : ITemplateCollectionIdProvider {
  // Inject FactionService to detect active faction at provider-call time.
  // FactionService.Current.Id is reliably set when the collection
  // service consults providers during its Load().
  public IEnumerable<string> GetTemplateCollectionIds() {
    yield return active == "Folktails"
        ? "NaturalResources.IronTeeth"
        : "NaturalResources.Folktails";
  }
}
```

**ID conventions differ per layer:**
- Template collections: `<Category>.<Faction>` — `NaturalResources.Folktails`, `Buildings.IronTeeth`, `Characters.Folktails`, `ModularShaftParts.<F>`, `Planes.IronTeeth`. Plus `Common`.
- Material collections: bare faction names — `Folktails`, `IronTeeth`, `Common`.
- Common collections appear multiple times (multiple specs with `CollectionId="Common"`); the service handles dedupe across `Common`, but **does NOT dedup template-name conflicts** within faction collections — both factions have `Dandelion` as a bare-name template, so loading both raises `TemplateMappingException` from `TemplateNameMapper.Load` when a name collision occurs.

**Always yield only the OTHER faction's id**, not both, to avoid the duplicate-name throw.

### Cross-faction *trees and bushes* work; crops don't (without more work)

Loading the cross-faction `NaturalResources` collection makes
`NaturalResourceFactory.SpawnIgnoringConstraints("Mangrove", coord)` return a
real, interactive Mangrove that beavers can chop. **But only if the
`PlantableSpec` and `GatherableSpec` are stripped from those cross-faction
templates first** — otherwise `PlantingToolButtonFactory` and
`GathererService` walk every plantable/yield, can't find a matching
planter / recipe in the active faction, and crash via `Single(...)` at
`BottomBarPanel.Load()`.

The strip is best done via Harmony postfix on
`TemplateCollectionService.Load()`. `AllTemplates` has a get-only
public surface but is a normal auto-property — write through the
`<AllTemplates>k__BackingField` field via reflection.

ModdableTimberborn's `ITemplateModifier` solves the same shape; we
replicate the pattern in-tree to avoid the dep.

**Crops are different.** Stripping `PlantableSpec` from a cross-faction
crop makes it instantiate but not render — the natural-resource
initialization path apparently hides/destroys the visible mesh when no
planter pairing exists. To make crops render *as entities* you'd need
to also load a cross-faction planter (FarmHouse), which then drags in
`Goods`, `Needs`, recipes, localization, etc. — a deep cascade. For
*decorative* crops, see the prefab-clone pattern below.

### `BackwardCompatibleTemplateNames` (silent-substitution gotcha)

Pre-faction-split saves used bare names (`Maple`, `Mangrove`) for
content that's now faction-keyed. To keep those saves loadable,
`TemplateSpec.BackwardCompatibleTemplateNames` allows alias resolution
through `TemplateNameMapper`.

**The trap:** if you ask `NaturalResourceFactory.SpawnNew(name, pos, bool)`
for a template that doesn't exist in the active faction, the mapper
silently returns *some other* template via the alias chain (e.g. asking
for `Maple` on Ironteeth got us an `Oak`). The `void` return type hides
the substitution; no exception, no log.

**Use `SpawnIgnoringConstraints` instead** — it returns the
`NaturalResource` (or `null` on missing), so callers can detect the
miss. Inspect `spawned.GameObject.name` to verify what was actually
instantiated.

### `ISpecService.GetBlueprint(name)` ≠ `TemplateNameMapper`

Two distinct lookup mechanisms:

- `ISpecService.GetBlueprint(name)` — addressed by *path* (e.g.
  `"FarmHouse.Folktails"`). Throws `ArgumentException("Blueprint not
  found at path: ...")` on miss. No alias / backward-compat resolution.
  Bare names like `"Maple"` don't resolve here.
- `TemplateNameMapper` — addressed by template name; consults
  `BackwardCompatibleTemplateNames`. Used internally by the natural-
  resource factory.
- `TemplateCollectionService.AllTemplates` — `ImmutableArray<Blueprint>`.
  Walk this to find a blueprint by `Blueprint.Name`. The bare-name
  index that `GetBlueprint` doesn't provide.

## Passive visual objects (prefab clone, no entity registration)

For purely-decorative ecology objects that should display Timberborn
art without participating in the entity system (no save/load entry, no
selection panel, no NavMesh blocking, no Cuttable/Plantable behaviour):

```csharp
public class PassiveObjectProbe : IPostLoadableSingleton {
  private readonly IPrefabOptimizationChain _chain;
  private readonly TemplateCollectionService _templates;

  public void PostLoad() {
    var donor = _templates.AllTemplates.First(b => b.Name == "Maple");
    var optimized = _chain.Process(donor);  // returns the cached optimized prefab
    var clone = UnityEngine.Object.Instantiate(
        optimized,
        CoordinateSystem.GridToWorld(tileCoord),  // un-centered for NR-style anchor
        Quaternion.identity);
    // Clone is now a free-standing GameObject; no entity registration.
  }
}
```

**What you get for free** with this pattern:
- Correct mesh + material (Timberborn shaders, lighting, day/night response).
- Idle animation (wind sway etc.) — runs on the cloned components.
- No entity registration → cannot be selected, cannot block paths,
  doesn't appear in the entity inspector. Beavers and builders ignore it.
- Multiple growth/state variants are baked into the prefab as child
  GameObjects — see hierarchy section below.

**What you own:**
- Save/load (the GameObject is gone after reload — regenerate from a
  deterministic seed, the DESIGN.md preference for decorative flora).
- Despawn (`UnityEngine.Object.Destroy(clone)`).
- Scene-transition cleanup (parent the clones to a scene-scoped object,
  or destroy explicitly when the Game scope unloads).

### Performance shape (measured Ryzen 9 + RTX 4070 Super)

Bulk-spawn stress test of 10000 cloned Canola prefabs:

| Layout | Result |
|---|---|
| Instantiation | ~30 µs per clone, 287 ms total for 10k |
| Stacked 100×100 on tile, all visible zoomed in | 10 fps — overdraw bottleneck |
| Stacked, zoomed out (sub-pixel) | 40 fps — overdraw collapses |
| Spread 100×100 grid (1/tile), zoomed in | 60 fps cap — basically free |
| Spread, zoomed out (all 10k in frustum) | 40 fps |
| 10k clones with `SetActive(false)` | no measurable FPS impact |

Findings:

- **Wind sway is a vertex shader, not an Animator.** No `Animator`
  component anywhere in the Carrot/Canola hierarchy; the prefab's wind
  sway is computed on the GPU per vertex. Animation cost scales with
  visible vertex count, not instance count — essentially free per
  CPU-side instance.
- **Per-instance CPU cost is negligible at scale.** Stripping
  components (NaturalResource, BlockObject, Cuttable, Growable…)
  produced no measurable improvement. The dormant components don't
  register with the entity system and don't run per-frame.
- **Render cost is dominated by overdraw + visible-instance count.**
  `SetActive(false)` on the entire pool eliminates the render cost
  fully; the 10k inactive GameObjects sit in the scene at no cost.
- **The pool model works.** Spawn N up-front, toggle Active based on
  whatever culling logic, only the active set burns frame budget.

**Density budget for production decoration.** From this hardware,
~3000 visible decorative prefabs at once is comfortable; ~10000 is
the hard ceiling at 40 fps. Worst case is **max-map zoomed all the
way out** (256×256 = 65k tiles fully visible) — at one decoration per
tile this would tank to ~5 fps. So density has to be a real design
parameter:

- Sparse default density (~1 per 4-16 tiles), biome-driven.
- Distance-from-camera culling beyond N tiles (toggle Active).
- Chunk-level mesh combination for very dense tiers (one draw per
  chunk instead of N).
- Detail-tier separation: full prefabs for featured items, instanced
  billboards or `Graphics.DrawMeshInstanced` for grass-class detail.

For Phase 1 prototype, the simplest path is a tunable density cap
fed by the surveyor; we tune the number to feel right on a max map.

## Particle / atmospheric effects

Timberborn's in-game atmospherics (smoke from chimneys, mist around
badwater, fluid spray) are **not** exposed as Unity's standard
`ParticleSystem` / `VisualEffect` / `TrailRenderer` /
`LineRenderer` components on the optimized prefabs we can reach via
`IPrefabOptimizationChain.Process`. A scan across all 200+ blueprints
came back empty for those component types and for any class name
containing "Particle", "Effect", "Smoke", "Mist", "Fog", "Emitter",
or "Steam". Most likely they're either stripped during prefab
optimization, custom-rendered through a separate service, or
implemented as shader-based UV scrolling on billboards (same
"animation in the shader" trick as the wind sway).

**Implication:** to add custom atmospherics (mist, fish, flying
critters, ground critters), we author our own assets via the Unity
SDK rather than cloning Timberborn ones.

### Code-built `ParticleSystem` works for integration testing

Constructing a `ParticleSystem` programmatically — `AddComponent` to
a fresh GameObject, configure via `.main`/`.emission`/`.shape`/
`.velocityOverLifetime` modules — works fine and validates the
spawn/lifecycle pattern without authored assets. Visuals render
**magenta** because Unity's default particle material's shader isn't
compatible with Timberborn's render pipeline (URP), the same fallback
we got from `GameObject.CreatePrimitive`. Geometry, motion, and
emission rates are correct; only the material is broken.

This pattern lets us prototype the *integration* of atmospherics
(positioning, lifecycle, scoping to scene) before SDK-authored
materials replace the magenta defaults. Substituting an SDK prefab
later is one line — `Instantiate(loadedPrefab, pos, rot)` instead
of `AddComponent<ParticleSystem>()`.

**Coordinate-convention quirk:** natural-resource prefabs use the
`BlockObject` anchor convention (origin at the tile's south-west
corner). Use `CoordinateSystem.GridToWorld(coord)` for them.
Unity primitives via `GameObject.CreatePrimitive` have their pivot at
the geometry center — for those, use `GridToWorldCentered`. Mixing the
two leaves the visual offset by half a tile.

        ...
      #Dying
        ...
    <Stage2>           # e.g. Mature
      #Alive
        ...
      #Dead
      #Dying
```

For crops the stages are typically `Seedling` and `Mature`; trees use
the `Growable` lifecycle and have analogous stage variants. The state
variants (`#Alive`, `#Dead`, `#Dying`) let us reflect ecological state
visually without running the full simulation — set `gameObject.SetActive(true)`
on the variant matching the chunk's current condition.

### `IPrefabOptimizationChain.Process(Blueprint)`

The DI-injectable handle on the prefab pipeline:

```
GameObject Process(GameObject)        // takes raw prefab, returns optimized
GameObject Process(Blueprint)         // takes blueprint, returns optimized
ImmutableArray<GameObject> GetCached() // already-processed prefabs
```

`Process(Blueprint)` resolves the blueprint's prefab through the
optimization chain (mesh merging, material atlas binding, etc.) and
returns the cached result. The returned `GameObject` is the *cache
entry* — clone with `UnityEngine.Object.Instantiate` rather than
modifying it directly.

## `NaturalResourceFactory` call surface

```
NaturalResource SpawnIgnoringConstraints(string, Vector3Int)     // safe, null on miss
NaturalResource SpawnIgnoringConstraintsAndRandomizePosition(string, Vector3Int)

| Method | Semantics | On missing template |
|---|---|---|
| `SpawnIgnoringConstraints` | Returns the entity. Skips placement validation. | Returns `null`. Safe. |
| `SpawnIgnoringConstraintsAndRandomizePosition` | Same, with intra-tile position jitter. | Returns `null`. |
| `SpawnNew` | Goes through the full validate-and-instantiate path. The `bool` parameter is poorly understood — *not* "spawn as adult". | **Silently substitutes** via `BackwardCompatibleTemplateNames`. Avoid. |

### Adult-spawn via `Growable._timeTrigger`

Spawned natural resources start at growth=0 (seedling/sapling). To make
a tree appear adult immediately:

```csharp
var growable = resource.GetComponent<Growable>();
var field = typeof(Growable).GetField(
    "_timeTrigger", BindingFlags.Instance | BindingFlags.NonPublic);
var trigger = (ITimeTrigger)field.GetValue(growable);
trigger.FastForwardProgress(trigger.DaysLeft);
```

`Growable` exposes nothing public; the private `_timeTrigger` is an
`ITimeTrigger` (public interface). `FastForwardProgress` is the same
path the game uses when growth completes naturally — fires the
finished event chain correctly, swaps to the adult mesh.

**Performance note:** reflection per spawn is fine for one-shot use.
For mass-spawn (hundreds of trees on game start), cache the
`FieldInfo`, or compile a typed delegate via `FieldInfo.CreateDelegate`
for ~10ns direct field access.

## `TemplateCollectionService.AllTemplates` is writable via reflection

The autogen shows `AllTemplates` as a get-only `ImmutableArray<Blueprint>`,

## Blueprint construction at runtime

Tested empirically (see `StrippedEntityProbe` and `DerivedBlueprintProbe`
in `src/Keystone.Mod/Debug/`).

**Bottom line.** Code-built blueprints via ctor #1 *do* work, as long
as you pass the donor's `Children`. Ctor #2 (spec replacement) is a
trap. The earlier "SDK is required" verdict was based on a self-inflicted
bug where we passed empty children and dropped the visual sub-tree.

### How vanilla blueprints are wired

Vanilla flora blueprints don't carry a `TimbermeshSpec` at the root.
They carry a `BlockObjectModelSpec` whose fields are *string names*:
`FullModelName` (typically `#Models`), `UncoveredModelName`,
`UndergroundModelName`. The spec doesn't bring the visual; it tells
`BlockObjectModel.Awake` where to find it.

The visual lives in **nested child blueprints** (Blueprint.Children).
A vanilla flora's Children include `#Models` whose specs include a
`TimbermeshSpec` pointing at the asset-bundle `.timbermesh`. When
`BlueprintPrefabConverter.Convert` walks the blueprint tree, each
child becomes a child GameObject named for its dictionary key. The
three `ISpecToPrefabConverter`s (`TimbermeshSpec`, `CollidersSpec`,
`TransformSpec`) attach the per-child mesh, colliders, and transform.

Result: `BlockObjectModel.Awake` finds `#Models` because we built
that child from spec data, not because a `.prefab` asset was loaded.

See `Assets/Mods/ShantySpeaker/Data/Buildings/.../ShantySpeaker.IronTeeth.blueprint.json`
in the SDK for a worked example — the `Children` block at the bottom
is exactly this nested structure.

### Ctor #1: `new Blueprint(name, specs, children)` — works

```csharp
var stripped = new Blueprint(
    name: "Keystone.Stripped." + donor.Name,
    specs: donor.Specs.Where(s => !stripList.Contains(s.GetType().FullName)),
    children: donor.Children);  // CRITICAL -- carries the visual sub-tree
```

This produces a working entity that:
- Renders correctly (vanilla timbermesh reused via inherited Children)
- Reacts to soil moisture via the kept `WateredNaturalResourceSpec` and
  the decorator chain that triggers `WateredNaturalResource`
- Self-deletes after `DaysToDieDry` of dry soil
- Can have specific behaviors stripped — Gatherable, Plantable,
  Yielder, Cuttable, etc. — without affecting the rest

Some decorators stay attached even when their "matching" spec is
stripped (e.g. `Demolishable` is wired to a foundational spec, not
its own `DemolishableSpec`). Stripping its config spec without removing
the decorator causes `Awake` null-derefs. Verify each strip empirically.

### Making an entity ambient (no select, no demolish)

`SelectableObject` is wired to a foundational spec (likely
`BlockObjectSpec` or `NaturalResourceSpec`) we must keep for tile
placement and moisture reactivity, so we can't get rid of it via spec
stripping. None of these less-invasive routes work either:

- `BaseComponent.DisableComponent` — the retriever finds disabled
  components anyway.
- Unity collider disabling — selection is tile-based, not raycast.
- `.enabled = false` — doesn't exist on `BaseComponent` (it's not a
  `UnityEngine.Behaviour`; only `BaseComponent.Enabled` exists with
  capital E, and that's what `DisableComponent` already toggles).
- `ComponentCache.RemoveDisabledComponent` via reflection — name is
  misleading. It removes from one of the cache's secondary tracking
  sets but **leaves the component in the master `_components` list**;
  `GetComponent<T>()` still finds it and so does the retriever.

What works: **Harmony patches keyed off the blueprint-name prefix.**
Three patches are needed in tandem to fully isolate the entity from
player interaction:

1. `SelectableObjectRetriever.TryGetSelectableObject(GameObject, out SelectableObject)`
   prefix → return false, set out param to null. Blocks cursor
   *hover* highlight (the raycaster path).
2. `EntitySelectionService.Select(BaseComponent)` prefix → skip when
   the target's GameObject name has our prefix. Required because
   `Select` calls `GetSelectableObject(target)` which is assertion-
   style: if our retriever patch returns false from inside, it throws
   "SelectableObject component not found". Patching `Select` short-
   circuits the click flow before that exception path.
3. `DemolishableSelectionTool.ActionCallback` and `PreviewCallback`
   prefixes → filter the `IEnumerable<BlockObject>` parameter to
   exclude entities with our prefix. Blocks the demolish-tool's
   preview ghost *and* the commit step. `Demolishable` is wired
   like `SelectableObject` (foundational spec, can't be cleanly
   stripped or DisableComponent'd off), so the tool's input is the
   right intercept level.

See `src/Keystone.Mod/HarmonyPatches/`:
- `SelectableObjectRetrieverPatch.cs`
- `EntitySelectionServicePatch.cs`
- `DemolishableSelectionToolPatch.cs`

The filter is strict prefix match on the blueprint name
(`Keystone.Stripped.`), so vanilla entities are untouched.

The blueprint's name is the *toggle*: name a blueprint
`Keystone.Stripped.<X>` and you get the ambient (unselectable,
undemolishable-by-player) variant; omit the prefix and you get the
selectable-doodad variant. Same construction code path for both.

**What still works.** Code-driven removal — `EntityService.Delete`,
`WateredNaturalResource` auto-delete after `DaysToDieDry`, anything
Keystone explicitly invokes — bypasses all three patches because
they're player-tool-specific. The mod always retains control over
its own entities.

### Walk-through and build-over (full ambient behaviour)

Two additional knobs at construction / post-spawn time complete the
"ambient" feel:

- **`BlockObjectSpec.Overridable = true`** via the record `with`
  syntax during spec assembly. The placement validator then permits
  buildings on top of our flora; the standard natural-resource
  clearing flow auto-removes the entity when the build commits.
- **Disable `BlockObjectNavMesh` (and `BlockObjectPreviewNavMesh`)**
  on the spawned entity. Both are internal types; we find them by
  walking `BaseComponent.AllComponents` and matching on type name
  (`GetComponent<T>()` won't compile for internal T). Disabling them
  removes the entity's tile from the navmesh's "blocked" set so
  beavers walk through unhindered.

Combined with the three Harmony patches, the entity is invisible to
all player interaction *and* doesn't impede gameplay (building or
movement). It's purely a visual + ecology participant.

### Bulk-demolish injection: re-exposing ambient entities to one specific tool

Counterpart to "Making an entity ambient" above. Once an entity is
fully ambient (no hover, no click, no resource demolish), getting it
back onto a *specific* deletion path requires understanding how the
target tool gates its input. The case Keystone solves: Class B
mini-flourishes should not be individually selectable, but the
**building bulk-demolish tool** (drag a rectangle, instant deletion
of everything inside) should sweep them up alongside real buildings.

Findings below are from decompiling
`Timberborn.DeconstructionSystemUI.dll`,
`Timberborn.BlockObjectTools.dll`, and
`Timberborn.AreaSelectionSystem.dll` via `ilspycmd` against the local
1.0 install. The `decomp/` folder in this repo is gitignored — regen
on demand with:

```powershell
ilspycmd "<install>\Timberborn_Data\Managed\Timberborn.<Asm>.dll" `
  --referencepath "<install>\Timberborn_Data\Managed" -o decomp
```

#### The two demolition tools and their split

| Tool | Class | Input filter | Deletion |
|---|---|---|---|
| Building bulk-demolish (instant) | `Timberborn.DeconstructionSystemUI.BuildingDeconstructionTool` extends `BlockObjectDeletionTool<BuildingSpec>` | runtime: `GetComponent<BuildingSpec>() != null` | `EntityService.Delete(bo)` per entity |
| Resource bulk-demolish (mark, beaver work) | `Timberborn.DemolishingUI.DemolishableSelectionTool` | `DemolishableSpec` + `Demolishing` runtime components | beaver `DemolishBehavior` |

The split is **type-based**, not flag-based. `DemolishableSpec.ShowDemolishButtonInEntityPanel`
controls only the entity-panel button visibility, not which path
fires. Switching between paths means changing what the entity is
(adding/removing `BuildingSpec`, `NaturalResourceSpec`, etc.), which
generally pulls in unwanted side effects.

#### The picker filter (the load-bearing claim)

`AreaBlockObjectAndTerrainPicker.GetBlockObjects<T>` runs a runtime
LINQ predicate on the picker output:

```csharp
return new PickingResult(
    from blockObject in _blockObjectPicker.PickBlockObjects(...)
    where blockObject.GetComponent<T>() != null
    select blockObject,
    ...);
```

For the building tool, `T = BuildingSpec`. Entities without a runtime
`BuildingSpec` component are dropped here, **before** reaching either
`PreviewCallback` or `ActionCallback`. This is what makes the spec
the discriminator.

#### Patch sites and shape

Both `PreviewCallback` (per-frame highlight) and `ActionCallback`
(commit) receive the entity list as
`(IEnumerable<BlockObject> blockObjects, IEnumerable<Vector3Int> terrainBlocks, Vector3Int start, Vector3Int end, bool selectionStarted, bool selectingArea)`.
A Harmony Prefix can declare `ref IEnumerable<BlockObject>
blockObjects` and rewrite the parameter — the mutation propagates to
the original method's body. (This works for non-`ref` reference-type
parameters.)

The injected entities don't need to satisfy any later check inside
the tool: the deletion path
(`BlockObjectDeletionTool<T>.DeleteBlockObjects`) is fully generic
over `BlockObject` and just calls `_entityService.Delete(bo)`. There
is no `GetSpec<T>()` lookup inside deletion; the generic parameter
is consumed only by the picker filter and by per-class tooltip /
dialog content (which gracefully ignore unknown entity shapes).

`ActionCallback` is private on the open generic
`BlockObjectDeletionTool<T>`, but Harmony's
`[HarmonyPatch(typeof(BlockObjectDeletionTool<BuildingSpec>),
"ActionCallback")]` resolves the closed generic directly. Other
subclasses (`EntityBlockObjectDeletionTool` over `EntityComponent`,
the recovered-good deletion tool over `RecoveredGoodStack`) use
unrelated generic args and stay untouched.

`PreviewCallback` in `BuildingDeconstructionTool` is a `protected
override` that pre-processes the list (adds recoverable-good
companions, etc.) before delegating to the base. Patching the
override means the injection happens before that pre-processing
runs, so the highlight covers our entities too.

#### Auxiliary surfaces verified safe

Walking through `BuildingDeconstructionTool` and its base:

- `BuildingDeconstructionTool.FillObjectsToDeconstruct` queries each
  entity for `IRecoverableObjectAdder`. Entities without it
  contribute nothing to the recoverable-good tooltip. Safe.
- `BlockObjectModelBlockadeIgnorer.IgnoreModelBlockades` iterates
  `BlockObjectSpec`. Class B has it; safe.
- `SetVisibleLayerToShowAllObjects` reads
  `PositionedBlocks.GetAllCoordinates()`. Generic over
  `BlockObject`; safe.
- `_undoRegistry.CommitStack()` runs after deletion — but the only
  concrete `IUndoRegistry` in the entire 490-DLL surface is
  `Timberborn.GameScene.DummyUndoRegistry`, a stub. Timberborn 1.0
  ships no player-facing undo, so this is a no-op. **Don't rely on
  undo working** in any feature design (a likely mistake when reading
  the API surface and seeing the full `Undo()` / `Redo()` /
  `CanUndo` / `CanRedo` shape on `IUndoRegistry`).

#### What doesn't work (and why)

- **Patching `IsBlockObjectValid`** (a `protected virtual` per-entity
  predicate). Runs *after* the picker filter, so the entity list it
  sees already excludes Class B. Can only narrow, not widen.
- **Patching `Enumerable.Where` directly** (the LINQ predicate at
  `GetBlockObjects`). Too brittle; Harmony Transpilers on LINQ
  internals break across .NET updates.
- **Postfix on `GetBlockObjects<T>`** to extend the
  `PickingResult.BlockObjects` enumerable. The result type is a
  `private readonly struct` — can't be constructed externally.

The Prefix on the callbacks is the cleanest cut.

#### Where this leaves the demolish landscape

| Player input | Class A | Class B | Class C | Class D |
|---|---|---|---|---|
| Hover highlight | n/a (non-`BlockObject`) | suppressed | normal | normal |
| Click select | n/a | suppressed | normal | normal |
| Resource demolish (mark + beaver) | n/a | excluded | included | included |
| Building bulk-demolish (instant) | n/a | **included via this patch** | excluded | excluded |
| `EntityService.Delete` from code | applicable | applicable | applicable | applicable |

Class B is now the only class reachable through the building bulk-
demolish path. Other classes stay where they were.

### Ctor #2: `new Blueprint(donor, originalSpec, newSpec)` — trap

"Replace one spec with another." Empirically:
- Preserves `donor.Name` (so the prefab cache hits → visual hierarchy
  is there → no `#Models` crash)
- Returns a *distinct* Blueprint object (`ReferenceEquals(derived, donor)
  == false`)
- But `DyingNaturalResource.Awake` throws `InvalidCastException` from
  `ComponentCache.GetCachedComponents<T>` — even with a no-op
  replacement (same spec instance for both arguments)

The architecture isn't built for two distinct Blueprint objects with
the same name to coexist. The per-instance component cache type-confuses
between them. Don't use ctor #2 for derive-from-vanilla; use ctor #1
with a renamed blueprint and pass `donor.Children`.

### When the SDK / Blender plugin is needed

Code-built blueprints can reuse vanilla timbermeshes by inheriting
`donor.Children`, but novel visuals have to come from somewhere.
Mesh authoring lives in the **Blender plugin**, not the Unity SDK
(see "Custom mesh authoring" below). The Unity SDK is for the
adjacent concerns:

- Asset bundles for materials, UI sprites, shaders
- Hand-tuned `BoxCollider`/`SphereCollider`/`CapsuleCollider`
  authoring (via `CollidersSpec` in the blueprint Children)
- Localization CSV
- The Mod Builder window that copies `Data/` and built DLLs to
  `%USERPROFILE%\Documents\Timberborn\Mods\<mod>`

So: **Blender plugin** for mesh authoring, **Unity SDK** for
asset-bundle packaging + deploy, **code-built blueprints** for
configuration variants (different `DaysToDieDry`, different
`BlockObjectSpec` sizes, etc., all sharing one set of vanilla
visuals). All three compose; you don't have to use them all.

## Custom mesh authoring (timbermesh pipeline)

Distilled from the Mechanistry wiki + plugin manual + the
`timberbornexamplemodels.blend` reference file. Add notes here as we
hit non-obvious behaviour during actual authoring.

### Tooling

- **Blender plugin repo:** https://github.com/mechanistry/timbermesh
  (separate from the main `timberborn-modding` repo). Plugin manual:
  https://github.com/mechanistry/timbermesh/wiki/Timbermesh-Blender-Plugin-manual.
- **Format:** Protocol Buffers-based. AssetRipper/TimberRipper cannot
  decode `.timbermesh` to a usable mesh — its GLB exporter sees the
  prefab tree but the geometry lives inside an opaque `_bytes` field
  on a `Timberborn.TimbermeshEditorTools` MonoBehaviour. Don't go
  down that road for inspection; use the Blender reference pack
  instead.
- **Blender version:** 2.80+.
- **Reference asset pack** (vanilla `.blend` files for inspection
  and modification): linked from
  https://github.com/mechanistry/timberborn-modding/wiki/Assets.
  Currently contains five examples in one scene: RooftopTerrace,
  AquaticFarmhouse, Smelter, PineTree (with seedling/dry/dead
  variants), Sunflower (with variants). Two flora examples cover
  the lifecycle-state hierarchy we care about for ambient flourishes.

### Export workflow

1. Author or modify a Blender collection.
2. Right-click the collection in the Outliner → **Export
   (\*.timbermesh)**. (Multiple collections: Ctrl+click, then
   **Batch Export (\*.timbermesh)**.)
3. Drop the resulting `.timbermesh` into the mod's `Data/...` tree
   so its name matches the path referenced from the blueprint's
   `TimbermeshSpec.Model` (without the extension).
4. Run the Mod Builder in Unity to copy it to the deploy folder.
   No transformation happens at this step — the Mod Builder treats
   `.timbermesh` as opaque bytes, copy-through.

### Two `#` conventions — same character, different scopes

It's tempting (and we got this wrong once) to assume the `#`
prefix means one thing across the whole pipeline. It doesn't.

- **Blender object names: `#` = "don't merge with siblings."**
  By default the exporter merges sibling meshes for performance.
  Prefixing an object's name with `#` keeps it as a separate
  sub-object inside the resulting `.timbermesh`. This matters
  when one `.timbermesh` carries multiple parts that need
  independent animation, visibility, or transform — typically
  buildings with moving pieces.
- **Blueprint Children keys: `#` = state-machine node.** Inside
  a blueprint's `Children` dictionary, keys starting with `#`
  (`#Models`, `#Alive`, `#Dying`, `#Dead`) name the state-machine
  nodes the blueprint→prefab converter creates as GameObject
  children at load time. Each leaf node has its own
  `TimbermeshSpec.Model` pointing at a (separate) `.timbermesh`
  file. Lifecycle code looks these names up by string to swap
  visibility.

The two conventions only collide when a single `.timbermesh`
needs to host multiple lifecycle variants. Vanilla doesn't do
that — each variant is its own file — so the `#` prefix never
appears Blender-side for vanilla flora. Look at the
`timberbornexamplemodels.blend` PineTree collection: meshes are
named `PineSeedling`, `PineSeedlingDry`, `PineMature`, ... — no
hashes. Hashes only appear in the *blueprint* JSON where the
state-machine hierarchy is wired.

Practical consequence for ambient flourishes: in Blender, name
the variant meshes whatever's clear (e.g.
`KeystoneFlourishTest_Alive`, `KeystoneFlourishTest_Dry`). Export
each as its own `.timbermesh`. Reference each one from the
appropriate `#Alive`/`#Dying`/`#Dead` node of the blueprint
Children. No `#` prefixes in the Blender file unless you have a
specific anti-merge need.

### Animation modes

- **Vertex animation** (Vertex Animation Texture / VAT): mesh-only,
  bakes per-vertex motion into a texture. Memory-heavy but works
  inside the merged-mesh batching path.
- **Node animation**: works on EMPTY *and* MESH objects, lighter
  weight, but the animated nodes can't be merged (so they need a
  `#` prefix or equivalent isolation).

For static flourishes neither mode applies; we just export the
geometry.

### Material naming

Confirmed against `timberbornexamplemodels.blend`'s PineTree
collection: the convention is **one material per species, named
after the species itself, shared across every lifecycle variant**.
PineTree's six variant meshes (`PineSeedling`, `PineSeedlingDry`,
`PineSeedlingDead`, `PineMature`, `PineMatureDry`,
`PineMatureDead`) all reference a single material literally named
`Pine`. No `Base*` prefix, no `_Green`/`_Dry` suffixes.

This is settled by **`IMaterialRepository.GetMaterial(string)`** at
runtime — the bare material name from the `.timbermesh` is the
lookup key. The runtime swaps in the actual textured shader.

**Implication for authoring**: lifecycle visuals come from
**geometry swaps, not material swaps**. The dying-vs-alive look
is baked into different mesh data (drooping leaves, browner
texture map painted into the geometry's UVs, whatever); the
material reference is constant. So when authoring a
Keystone-original flourish, build N variant meshes, give them
all the same material, name the material after your species
(e.g. a `Cattail`-style flourish would have material name
`Cattail` — though we'd want a Keystone-prefixed name for any
original species we ship to avoid colliding with vanilla).

If a species we invent doesn't have a corresponding runtime
material, `GetMaterial` will presumably either fall back or
fail loudly — TBD; first time we ship a custom material name
we'll see which.

## Animated meshes via VAT

Distilled from the `KeystoneDeer` smoke test (Quaternius FBX →
Blender → `.timbermesh` with VAT → in-game animated playback).
This pipeline is **not documented end-to-end anywhere**; the
wiki covers Blender export but not the Unity-side shader
requirement, and the IAnimator interface has a couple of
non-obvious gotchas. Capture here so the next animal import
doesn't relearn it from scratch.

### How VAT actually works

Vertex Animation Texture: the per-frame post-armature vertex
positions of a deforming mesh get baked into a texture (one
texel per (vertex, frame)). At runtime, the vertex shader
samples the texture using `(vertexIndex, animationTime)` to
get the deformed position for the current frame, replacing
the mesh's static vertex buffer for that frame.

This means **VAT requires a shader that knows about it**.
URP Lit reads vertex positions from the mesh buffer only —
animated `.timbermesh` files render as static T-pose under
URP Lit, regardless of whether the `IAnimator` is "playing".

The sampling helper lives at
`Assets/Tools/ImportedAssets/Editor/Resources/Shaders/AnimateVAT.hlsl`
in the SDK; vanilla animated entities (beavers, etc.) use
shaders that call this function in their vertex stage. The
beaver's shader is `Shader Graphs/BeaverURP` (GUID
`f39ae455f01c34d4b9c2a3ca9e2f66ec`) and exposes properties
`animationTime` / `offsets` / `rotations` — the Timbermesh
importer auto-populates these from the VAT-baked
`.timbermesh`.

For a Keystone-original VAT material, two choices:
- **Quick**: point the mod's `.mat` at `BeaverURP` directly
  (works, but ties us to a vanilla asset path and inherits any
  beaver-specific inputs). `KeystoneDeer.mat` does this today.
- **Proper**: author a new ShaderGraph in the mod's
  AssetBundles tree that calls `AnimateVAT.hlsl` via a Custom
  Function node, with `animationTime`/`offsets`/`rotations`
  properties named exactly so the importer wires them up. Same
  shape as BeaverURP but without the fur-tint inputs.

### Blender exporter behaviour for animations

The plugin's `animation_builder.py` confirms two non-obvious
rules (verified against the live source at
`github.com/mechanistry/timbermesh/src/timbermesh_blender_plugin/`):

**Action selection is NOT selection-based.** The exporter
iterates `bpy.data.actions` and exports every Action stored
in the .blend file when "Single animation" is off. The active
action on the armature, what's highlighted in the Action
Editor, NLA strips, Fake User flags — none of these filter the
export. To exclude an Action, **delete it from the .blend or
remove its users** so Blender's GC drops it on save.

**"Single animation" trap.** With this option **on**, the
exporter creates one clip named `"Default"` whose frame range
comes from the Scene's frame range, **not** the active
Action's keyframe range. Real-world consequence: if your
scene timeline is 1..250 (Blender default) but the Idle
action's keyframes only cover frames 1..30, the baked clip
runs for 250 frames — the first 30 animate, the remaining 220
hold the last pose, then loop. Visible symptom: animation
plays through the first portion, freezes for a beat, then
snaps back. Fix is either (a) shrink the scene frame range to
match the Action's, or (b) turn Single animation **off** and
let each Action export with its own native length.

With "Single animation" **off**, each `bpy.data.actions` entry
becomes a clip in the export, named after the Action
(`Idle`, `Walk`, `Death`, ...) with the Action's own frame
range. This is the right mode for any asset with multiple
clips, and the right mode for single-clip assets too unless
you want the cosmetic value of the name `"Default"`.

### `IAnimator` interface quirks

`Timberborn.TimbermeshAnimations.IAnimator` is the runtime
handle Timberborn gives you over a VAT-driven prefab. A few
things that aren't obvious from the type signature:

- **`Speed` is write-only.** No getter on the interface — you
  can `_animator.Speed = x` but cannot read it back. Cache the
  value yourself if you need it.
- **`Play(string, bool)` takes positional args**, not named.
  `Play(name, loop)` works; `Play(name, loop: true)` doesn't
  compile.
- **`Play` throws on unknown clip name.** The exception
  message is `"Animation <name> not found in <X> animator."`
  Wrap in `try/catch` if you can't guarantee the clip exists.
- **`IAnimatorController` is rarely attached.** Despite the
  type's existence (and its `AnimationNames` enumerator,
  which would be useful for clip-name discovery), `.timbermesh`
  imports we tested expose `IAnimator` but not
  `IAnimatorController`. So you cannot list available clip
  names at runtime — they have to be known at authoring time.
- **Different-length clips work automatically.** Calling
  `Play("Walk", true)` then `Play("Idle", true)` switches the
  active clip and `AnimationLength` updates per-clip. The
  shader uses normalised time within the active clip
  (`AnimateVAT.hlsl`'s `frameIndex` is `[0, 1]` over
  `frameCount` per clip), so loops are always correct.

Find the animator on a spawned prefab via
`GameObject.GetComponentInChildren<IAnimator>(includeInactive: true)`
from a `BaseComponent` (e.g. on `IInitializableEntity.InitializeEntity`).
Reference impl: `src/Keystone.Mod/Fauna/KeystoneFaunaAnimator.cs`.

### Game-speed integration

The animator runs at real-time pace by default; multiply its
`Speed` by `NonlinearAnimationManager.SpeedMultiplier` so the
game-speed slider affects animation. Subscribe to
`CurrentSpeedChangedEvent` (`EventBus.Register(this)` +
`[OnEvent]`-attributed handler) to update Speed when the
player changes time speed.

**Empirical caveat:** `CurrentSpeedChangedEvent` fires for
slider changes between speed levels but is not consistently
delivered for pause/unpause toggles, and the multiplier reads
as 0 while paused. If you set `_animator.Speed =
SpeedMultiplier` at spawn time while paused, the animator
will be stuck at Speed=0 even after the player unpauses. Use
a sensible non-zero floor (`Math.Max(speedMultiplier, 1f)`)
at init, or skip the multiplier entirely for prototype work.

### Spec wiring

`TimbermeshAnimatorControllerSpec` is the spec the runtime
ships for state-machine playback (`docs/timberborn-specs.md`
line 1828). The mapping `TimbermeshAnimatorControllerSpec`→
`TimbermeshAnimatorController` is auto-decorated. For simple
"always play one clip on loop" cases, that spec isn't needed
— a Keystone-side `BaseComponent` that calls
`_animator.Play(...)` directly is enough.

## Sub-tile composition via `TransformSpec` (blueprint-time)

Confirmed empirically against
`unity-assets/Keystone/Data/NaturalResources/KeystoneFlourishTest/`.

`Timberborn.UnityEngineSpecs.TransformSpec` (`Position`, `Rotation`,
`Scale` Vector3 fields) **works at the blueprint→prefab conversion
step**, before the rendering pipeline reads positions. That makes
it a real lever for sub-tile composition from JSON alone, with no
Blender authoring required. This contradicts the earlier
"transforms don't move flora" finding — see § "Visual position is
custom-rendered" below for the runtime/blueprint distinction.

### Working example

```jsonc
"Children": {
  "#Models": {
    "Children": {
      "Pine": {
        "TimbermeshSpec": { "Model": "NaturalResources/Trees/Pine/Mesh/PineMature" },
        "TransformSpec":  {
          "Position": { "X": 0.0, "Y": 0.0, "Z": 0.0 },
          "Rotation": { "X": 0.0, "Y": 0.0, "Z": 0.0 },
          "Scale":    { "X": 0.5, "Y": 0.5, "Z": 0.5 }
        }
      },
      "Sunflower": {
        "TimbermeshSpec": { "Model": "NaturalResources/Crops/Sunflower/SunflowerMature" },
        "TransformSpec":  {
          "Position": { "X": 0.5, "Y": 0.0, "Z": 0.5 },
          "Rotation": { "X": 0.0, "Y": 0.0, "Z": 0.0 },
          "Scale":    { "X": 0.5, "Y": 0.5, "Z": 0.5 }
        }
      }
    }
  }
}
```

That gives one entity, one tile, two visible sub-meshes (a
half-size Pine in the SW quadrant, a half-size Sunflower in the NE
quadrant), each pulling from vanilla bundles. No `.timbermesh`
authored.

### Coordinate convention: tile-centered origin, center pivot

`TransformSpec.Position` is **tile-centered** — `(0, 0, 0)` places the
mesh's visible centroid at the centre of the tile, not its SW corner.
The valid range across one tile is roughly `[-0.5, +0.5]` in X (east-
west) and Z (north-south). `+X` is east, `+Z` is north.

| Where | Position |
|---|---|
| Tile centre | `(0, 0, 0)` |
| NE quadrant centre | `(+0.25, 0, +0.25)` |
| NW quadrant centre | `(-0.25, 0, +0.25)` |
| SW quadrant centre | `(-0.25, 0, -0.25)` |
| SE quadrant centre | `(+0.25, 0, -0.25)` |
| Near NE corner | `(+0.5, 0, +0.5)` |

**Pivot is the mesh's geometric centre,** not its SW bounding-box
corner. Position is where the *visible centroid* lands. Verified
empirically across all vanilla mesh types we've tested -- crop
seedlings (Cattail, Cassava, Sunflower), bush seedlings
(BlueberryBush), and tree seedlings (Birch) all behave the same
way. (Earlier versions of this doc claimed SW-pivot + tile-corner
origin from a single Pine test that was misinterpreted; corrected
2026-05-10 against `KeystoneRiparianMini1..5` calibration blueprints.)

Rotations should stay at `(0, 0, 0)` for Keystone compositions — see
the project memory entry on no-rotations and the discussion in
`docs/biomes/riparian.md`. Pivot behaviour under non-zero rotation
hasn't been calibrated, and uncalibrated rotations have caused
hours of position-debug churn historically.

### Hard constraint: incompatible with `NaturalResourceSpec`

The hierarchy `#Models → { Pine, Sunflower }` only spawns cleanly
if **`NaturalResourceSpec` is absent**. With it present,
`NaturalResourceModel.ShowCurrentModel` runs at PostInitialize and
expects to find a renderer-bearing thing under the active state
node (`#Models → <stage> → #<state>`). Multi-children with
`TransformSpec` under a state leaf produces an NRE because the
model code doesn't walk into grandchildren to find a mesh.

So this composition style has a real architectural cost: the
entity is a plain `BlockObject`, not a `NaturalResource`. It's
invisible to `FloraCatalog`, has no moisture/contamination
lifecycle, isn't surveyed by ecology systems. It's purely
decorative.

### Required minimums

The smallest blueprint that spawns successfully via
`BlockObjectFactory.CreateFinished` and renders multiple sub-meshes:

- `TemplateSpec` (`TemplateName`)
- `BlockObjectSpec` (size, blocks, entrance — required for placement)
- `BlockObjectModelSpec` (points at `#Models`)
- `DemolishableSpec` — `Demolishable` is auto-attached to `BlockObject`
  and its `Awake` NREs without this spec. Set
  `ShowDemolishButtonInEntityPanel: false` to keep the panel button
  hidden; our Harmony patch suppresses the demolish tool itself.
- `Children` with `#Models` + sub-mesh nodes.
- (Keystone-specific) `KeystoneFlourishSpec` for the Harmony
  ambient-filter to recognise the entity.

Notably **no** `NaturalResourceSpec`, `WateredNaturalResourceSpec`,
`LabeledEntitySpec`, `BasicSelectionSoundSpec`,
`ContaminatedNaturalResourceSpec`, or
`NaturalResourceModelRandomizerSpec` — none of those are required
for a plain `BlockObject` decoration.

### When to use this vs the timbermesh path

| Need | Use |
|---|---|
| Visual decoration only, no ecology participation | This (blueprint composition) |
| Single asset, minor positional adjustment vs vanilla | This (one child + `TransformSpec`) |
| Multi-mesh + lifecycle reactivity (alive/dying/dead swaps) | Author one `.timbermesh` per state in Blender; one `TimbermeshSpec` per state node |
| Wholly new geometry | Author in Blender |

The blueprint-composition path covers a real product: ambient
flourishes that exist for visual richness without claiming a slot
in the simulation. Ecology-participating flora still goes through
the vanilla one-mesh-per-state pattern with Blender authoring.

## Blueprint overlay system (mod-time blueprint modification)

Reference: https://github.com/mechanistry/timberborn-modding/wiki/Blueprints#modifying-blueprints.

When a mod ships a JSON file at the same path as a vanilla
blueprint, the loader **merges** the two rather than overwriting.
Merge rules:

| Field syntax | Effect |
|---|---|
| `"Name": value` | Replace original value, or wholesale-overwrite a list |
| `"Name#append": [...]` | Append to existing list (deduplicated) |
| `"Name#remove": [...]` | Remove specified elements |
| `"Name#delete": null` | Delete the field entirely |
| Filename ends in `.optional.blueprint.json` | Only applied if the target blueprint exists; otherwise silently skipped |

When multiple mods overlay the same blueprint, **load priority
controls merge order** (later mods see earlier mods' merged
result).

### Where this fits in the extension-point ladder

CLAUDE.md prescribes "Bindito interface override → `MultiBind`
decorator → `EventBus` → lifecycle interface → Harmony last." The
overlay system slots in **before** Harmony for any "modify a
vanilla blueprint" need. It's purely declarative, version-stable
across game updates (as long as the spec class names hold), and
needs no DLL.

### Concrete Keystone applicability

- **Decorator attachment.** An overlay `Maple.optional.blueprint.json`
  with `"KeystoneFlourishSpec": {}` makes every vanilla Maple a
  flourish-eligible entity for our ecology layer — no Harmony
  patch on `TemplateCollectionService.Load`, no decorator
  registration gymnastics.
- **Spec tweaks.** `"WateredNaturalResourceSpec": { "DaysToDieDry": 3.0 }`
  changes vanilla flora's moisture sensitivity declaratively.
- **Strip behaviour.** `"PlantableSpec#delete": null` removes
  planting from a flora — would replace the body of
  `TemplateCollectionServicePatch` (the cross-faction strip) with
  a couple of overlay files.
- **Cross-cutting additions.** Adding
  `"ContaminatedNaturalResourceSpec": {}` to any flora that lacks
  it makes badwater interaction uniform across every plant.

### What overlays cannot do

- Add or remove decorators that aren't tied to a spec class —
  decorators are attached by the template chain when the spec is
  present, so `KeystoneFlourishSpec`-style additions work, but
  any decorator that's wired to a foundational always-present
  spec (`SelectableObject` on `BlockObjectSpec`, e.g.) can't be
  removed via overlay.
- Touch the entity's GameObject hierarchy — only the spec list
  and Children dictionary, both shaped by JSON.
- Change behaviour that lives in the C# decorators themselves.
  For that you still need code.

### Open question: composing overlay + Harmony

`TemplateCollectionServicePatch` currently runs as a Harmony
postfix on `TemplateCollectionService.Load`. If we also ship
overlays for the same blueprints, the order is overlay-load (game
side, before our Harmony postfix runs) → our postfix → final state.
We can collapse the strip work into overlays and retire that
patch when we've validated overlay timing covers all current
strip targets.

## Visual position is custom-rendered, not Transform-driven (at runtime)

Tested empirically (see `StrippedEntityProbe.RunStackTest` history in
git, plus the verbose hierarchy dumps captured in conversation logs).

**Important scope:** This finding applies to **runtime** Transform
manipulation only. **Blueprint-time `TransformSpec` works fine** —
see § "Sub-tile composition via `TransformSpec`" above for that
mechanism, which is the actually-supported path for sub-tile mesh
placement. The behaviour below describes what happens when you try
to override transforms *after* the entity is constructed, e.g. via
`LateUpdate`, reflection on a spawned entity, or direct
`transform.localPosition` writes from gameplay code.

**Vanilla flora rendering bypasses Unity's Transform hierarchy at
runtime.** You can write to `entity.GameObject.transform.localPosition`
(root, `#Models`, or even the leaf MeshRenderer-bearing nodes), watch
your value persist frame-to-frame in the Transform, and still see no
visible shift. The same is true for `localScale` and for direct
manipulation via reflection or per-frame `LateUpdate`.

### Evidence

- Per-frame `LateUpdate` re-asserting `localPosition = (0, 0, -0.4)` —
  observed value matches target on every read; visual remains centered.
- Offsetting the leaf MeshFilter-bearing transforms directly
  (`MapleSeedling`, `MapleMature`, etc.) — same null result visually.
- Hierarchy dump: **all** MeshRenderers (Seedling Alive/Dying/Dead,
  Mature Alive/Dying/Dead, Stump) have `enabled=true` simultaneously
  on a single entity, yet only one is visible. So `MeshRenderer.enabled`
  is not the gate either.

### Mechanism

`Timberborn.Rendering.MeshDrawer` exposes:

```
Void DrawMultipleInstanced(List<Matrix4x4> matrices)
Void Draw(Matrix4x4 matrix)
```

GPU instancing with explicit per-instance matrices. Whatever assembles
those matrix lists per-frame is the canonical source of "where this
flora draws." The matrix doesn't read from the Unity Transform — it's
populated from a different per-instance position holder, almost
certainly a chain through `NaturalResourceCenterProvider.GetWorldCenter()`,
`BlockObjectCenter`, and `CoordinatesOffsetter`.

### What can and can't be modified

- **`MarkerPosition`** (in `Timberborn.Rendering`) is for status icons /
  marker overlays above an entity, **not** the model's render position.
  Despite the suggestive name and `UpdatePosition(Vector3 gridOffset)`
  method, manipulating it doesn't shift the visible mesh.
- **`CoordinatesOffsetter`** can be `DisableComponent`'d but doing so
  has no observable effect on visible position post-Awake (its
  contribution may already be baked into the matrix at instance setup).
- **`NaturalResourceMarkerPositionUpdater`** (internal, in
  `NaturalResourcesUI`) updates `MarkerPosition` on model-change events.
  Disabling it changes nothing visible.

### What stack-tracing the suspects ruled out

We hooked Harmony postfixes that log stack traces on three suspect
methods to find the actual rendering pathway. Results:

- **`NaturalResourceCenterProvider.GetWorldCenter()`** is called only
  from gameplay code (`UncuttableYielding.UncuttableReacher.InitializeEntity`
  and `Forestry.TreeReacher.InitializeEntity`) — once per entity at
  init, never per-frame, never from a rendering path. It feeds
  reachability calculations, not the renderer. **Not the lever.**
- **`MeshDrawer.Draw(Matrix4x4)`** and **`MeshDrawer.DrawMultipleInstanced(List<Matrix4x4>)`**
  fire **zero times** for natural-resource flora. The `MeshDrawer`
  class isn't involved in flora rendering at all — it's used elsewhere
  (UI overlays, water, etc.). **Not the lever.**

Where flora actually renders from is still undetermined. Either some
other batched drawer we haven't traced, or Unity's standard
`MeshRenderer` pipeline (which would mean our `Transform` offsets
*should* have worked, and we may have misread the visual result).
Investigation paused; the path forward is SDK-authored content with
mesh vertices baked at desired sub-tile positions.

### Implication (revised after the `TransformSpec` finding)

**Per-instance *runtime* position override of vanilla flora is not
supported through any documented API.** The lever isn't there at
runtime. But we don't need it — `TransformSpec` at blueprint-build
time *is* supported, and that's enough for the sub-tile composition
use cases Keystone cares about. The runtime path is now interesting
mostly as a "don't do this" warning: don't write code that twiddles
transforms on spawned flora and expects visible movement; design the
position into the blueprint instead.

For Keystone, this means visual variation has two paths:

1. **Blueprint-time composition** (§ "Sub-tile composition via
   `TransformSpec`"): multi-child blueprints, vanilla mesh references,
   no Blender, no `NaturalResourceSpec`. Decoration-only flourishes.
2. **Authored `.timbermesh`**: when ecology-participating flora needs
   distinct geometry, author each mesh in Blender. One `.timbermesh`
   per (species, stage, state) slot, vanilla pattern.

The cross-mod compatibility story this implies: integrating with
third-party plant mods through the blueprint-composition path
means our curated flourishes can reference any mod's mesh asset by
path. The recipe still lives in our blueprint; only the geometry
comes from theirs. See `ROADMAP.md` § 1F for the architectural
framing of cross-mod content composition.

## Coordinate conventions

| Method | Pivot |
|---|---|
| `CoordinateSystem.GridToWorld(Vector3Int)` | Tile's south-west bottom corner |
| `CoordinateSystem.GridToWorldCentered(Vector3Int)` | Tile center on X/Z, bottom on Y |
| `CoordinateSystem.WorldToGridInt(Vector3)` | Inverse of `GridToWorld` |

`Vector3Int` tile coords are `(x, y, z)` where **z is the height
index**. Unity world `Vector3` is `(x, y, z)` where **y is up**.
`CoordinateSystem` handles the swap.

`BlockObject.Coordinates` is the entity's anchor tile (south-west
corner of its footprint). For 1×1 entities the anchor is the tile
itself; for multi-tile buildings the visual extends in +X / +Z from
there. This matches the prefab origin convention — that's why
`GridToWorld(coord)` is the right placement call for natural-resource
prefabs.
